Configuration

Spring Boot externalises configuration so the same JAR can run in development, staging, and production with different settings. Properties can come from files, environment variables, command-line arguments, or a cloud config server — each with a well-defined precedence order.

application.properties vs application.yml

Both formats are equivalent. YAML is preferred for structured config with many nested keys; .properties is simpler for flat key-value pairs.

# application.properties
server.port=8080
server.servlet.context-path=/api
spring.application.name=my-service

spring.datasource.url=jdbc:postgresql://localhost:5432/mydb
spring.datasource.username=admin
spring.datasource.password=secret
# application.yml — same values as above
server:
  port: 8080
  servlet:
    context-path: /api
spring:
  application:
    name: my-service
  datasource:
    url: jdbc:postgresql://localhost:5432/mydb
    username: admin
    password: secret

Property precedence (highest → lowest)

  1. Command-line arguments: --server.port=9090
  2. SPRING_APPLICATION_JSON environment variable
  3. OS environment variables: SERVER_PORT=9090
  4. Profile-specific files: application-prod.yml
  5. Default: application.yml / application.properties

Profiles

Profiles let you maintain different configurations per environment. Spring Boot activates profiles from the spring.profiles.active property.

# application.yml (shared base)
spring:
  jpa:
    show-sql: false

---
# application-dev.yml (dev overrides)
spring:
  config:
    activate:
      on-profile: dev
  datasource:
    url: jdbc:h2:mem:testdb
  jpa:
    show-sql: true
    hibernate:
      ddl-auto: create-drop

---
# application-prod.yml
spring:
  config:
    activate:
      on-profile: prod
  datasource:
    url: ${DATABASE_URL}
  jpa:
    hibernate:
      ddl-auto: validate
# Activate via command-line
java -jar app.jar --spring.profiles.active=prod

# Activate via environment variable
export SPRING_PROFILES_ACTIVE=prod

# Activate multiple profiles
java -jar app.jar --spring.profiles.active=prod,metrics

Reading individual properties

@RestController
public class InfoController {

    // Inject a single property value
    @Value("${spring.application.name}")
    private String appName;

    // With a default fallback
    @Value("${feature.dark-mode:false}")
    private boolean darkModeEnabled;

    // Inject the full Environment object
    @Autowired
    private Environment env;

    @GetMapping("/info")
    public Map<String, Object> info() {
        return Map.of(
            "app", appName,
            "profile", env.getActiveProfiles()
        );
    }
}

@ConfigurationProperties (recommended)

Bind a whole namespace of properties to a typed Java object. This is better than scattered @Value annotations because it is type-safe, IDE-assisted, and testable.

# application.yml
app:
  payment:
    provider: stripe
    api-key: ${STRIPE_API_KEY}
    timeout-seconds: 30
    retry-count: 3
@ConfigurationProperties(prefix = "app.payment")
@Validated
public record PaymentProperties(
    @NotBlank String provider,
    @NotBlank String apiKey,
    @Min(1) @Max(300) int timeoutSeconds,
    @Min(0) @Max(10) int retryCount
) {}

// Register it:
@SpringBootApplication
@EnableConfigurationProperties(PaymentProperties.class)
public class MyApp { ... }

// Or simply annotate the record with @Component

// Inject it:
@Service
public class PaymentService {
    private final PaymentProperties props;

    public PaymentService(PaymentProperties props) {
        this.props = props;
    }
}

Environment variables and secrets

Spring Boot automatically maps OS environment variables to property keys: DATABASE_URL → database.url (relaxed binding). Reference them in YAML with ${ENV_VAR}.

spring:
  datasource:
    url: ${DATABASE_URL}                # required — fails fast if absent
    password: ${DB_PASSWORD:}           # optional — empty string if absent
    username: ${DB_USER:app}            # with default value "app"
Never commit secrets. Use environment variables, a secrets manager (AWS Secrets Manager, HashiCorp Vault), or Spring Cloud Config Server for sensitive values. The ${VAR} syntax keeps secrets out of your source code.

Logging configuration

logging:
  level:
    root: INFO
    com.example.myapp: DEBUG          # verbose logging for your own package
    org.hibernate.SQL: DEBUG          # log Hibernate-generated SQL
    org.springframework.security: WARN
  file:
    name: logs/application.log
  pattern:
    console: "%d{HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n"
