Cluster Architecture

A Kubernetes cluster is split into two planes: the control plane, which makes decisions, and the worker nodes, which run your containers. Understanding what each component does makes everything else in Kubernetes — and every error message — far easier to reason about.

The big picture

Control Plane

  • kube-apiserver — the front door; everything talks to it
  • etcd — the cluster database
  • kube-scheduler — picks a node for each new pod
  • controller-manager — runs the reconciliation loops
  • cloud-controller-manager — talks to your cloud provider

Worker Node (× N)

  • kubelet — the node agent; starts and watches pods
  • kube-proxy — programs networking rules for Services
  • container runtime — containerd / CRI-O; runs containers
  • Pods — your application workloads

Every interaction — kubectl commands, controllers, kubelets reporting status — flows through the API server. Components never talk to each other directly; they all read and write objects in the API, which makes the system loosely coupled and observable.

Control plane components

kube-apiserver

The API server exposes the Kubernetes REST API. It validates requests, enforces authentication and authorization (RBAC), and is the only component that reads or writes etcd. When you run kubectl apply, you are making an HTTPS request to this component.

etcd

A consistent, distributed key-value store that holds the entire cluster state: every object you create, every status update. Lose etcd without a backup and you lose the cluster's memory — which is why production clusters run it in an odd-numbered quorum (3 or 5 members) and back it up regularly.

kube-scheduler

Watches for pods that have no node assigned and picks the best one for each. Scheduling runs in two phases:

  • Filtering — remove nodes that can't run the pod (not enough CPU/memory, missing labels required by a nodeSelector, taints the pod doesn't tolerate).
  • Scoring — rank the survivors (spread across zones, prefer less-loaded nodes, honor affinity rules) and bind the pod to the winner.

kube-controller-manager

A single binary that runs dozens of controllers — the reconciliation loops that give Kubernetes its self-healing behavior. Each controller watches one object type and drives actual state toward desired state. Examples:

  • Deployment controller — keeps ReplicaSets in line with the Deployment spec.
  • ReplicaSet controller — creates or deletes pods to match the replica count.
  • Node controller — notices unresponsive nodes and evicts their pods.
  • Job controller — runs pods to completion and tracks retries.
The reconciliation loop is the heart of Kubernetes: observe current state → diff against desired state → act to close the gap → repeat forever. Every controller follows this pattern.

cloud-controller-manager

Bridges the cluster to a cloud provider's APIs: provisioning load balancers for Service type: LoadBalancer, attaching cloud disks, and syncing node metadata. Local clusters (minikube, kind) simply don't run it.

Node components

kubelet

The agent on every node. It watches the API server for pods assigned to its node, tells the container runtime to start their containers, runs liveness/readiness probes, and reports pod and node status back. The kubelet only manages containers created through Kubernetes — it ignores containers you start manually.

kube-proxy

Implements the virtual IPs behind Services. It programs iptables or IPVS rules so traffic addressed to a Service IP is transparently forwarded to one of the healthy backing pods — no user-space proxying on the data path.

Container runtime

The software that actually pulls images and runs containers — typically containerd or CRI-O. Kubernetes talks to it through the Container Runtime Interface (CRI), which is why Docker Engine itself is no longer required on nodes.

Follow a kubectl apply through the system

Tying it together — here is what happens when you deploy an app:

  1. kubectl apply -f deploy.yaml sends the Deployment to the API server, which validates it and stores it in etcd.
  2. The Deployment controller sees the new object and creates a ReplicaSet; the ReplicaSet controller creates pod objects — still unscheduled.
  3. The scheduler notices pods with no node, filters and scores nodes, and binds each pod to one.
  4. The kubelet on each chosen node sees a pod assigned to it, asks the container runtime to pull the image and start containers, and reports status back.
  5. kube-proxy updates networking rules if a Service selects these pods; CoreDNS resolves the Service name.

Inspect the moving parts on your own cluster:

# Control plane components run as pods in kube-system
kubectl get pods -n kube-system

# Detailed view of a node and its capacity
kubectl describe node <node-name>

# Watch the event stream — the cluster narrating its own actions
kubectl get events --sort-by=.metadata.creationTimestamp
