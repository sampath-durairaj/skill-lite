Scaling & Autoscaling

Kubernetes can scale on three axes: more pods (horizontal), bigger pods (vertical), and more nodes (cluster). This page covers manual scaling, the Horizontal Pod Autoscaler that most teams rely on daily, and how the pieces fit together.

Manual scaling

kubectl scale deployment/web --replicas=10

# or declaratively — change replicas: in the YAML and
kubectl apply -f deployment.yaml

Manual scaling is fine for predictable changes ("scale up before the product launch"). For everything else, let the cluster react to load itself.

Horizontal Pod Autoscaler (HPA)

The HPA adjusts a Deployment's replica count automatically to hold a target metric — most commonly average CPU utilization — at a set level:

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 2
  maxReplicas: 15
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70    # aim for 70% of each pod's CPU *request*
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300   # wait 5 min before scaling down
kubectl apply -f hpa.yaml
kubectl get hpa web --watch
NAME   REFERENCE        TARGETS       MINPODS   MAXPODS   REPLICAS
web    Deployment/web   cpu: 43%/70%  2         15        3

How the math works

Every 15 seconds the HPA computes:

desiredReplicas = ceil( currentReplicas × currentMetric / targetMetric )

Example: 4 replicas averaging 90% CPU with a 70% target:
         ceil(4 × 90 / 70) = ceil(5.14) = 6 replicas
  • Utilization is measured against the pod's CPU request — an HPA on a Deployment without CPU requests does nothing but report errors.
  • Requires metrics-server (on minikube: minikube addons enable metrics-server).
  • Scale-up is fast; scale-down is deliberately slow (stabilization window) to avoid flapping.
Don't set replicas: in a Deployment managed by an HPA. Every kubectl apply would fight the autoscaler by resetting the count. Omit the field and let the HPA own it.

Beyond CPU

The autoscaling/v2 API also supports memory, multiple metrics at once (the largest result wins), and custom/external metrics — requests-per-second from your ingress, queue depth from RabbitMQ or SQS — via adapters such as Prometheus Adapter or KEDA, which can even scale workloads to zero.

Vertical Pod Autoscaler (VPA)

The VPA resizes pods instead of adding more — it observes actual usage and adjusts CPU/memory requests. It's an add-on (not built in), and its most popular mode is the safest one:

  • updateMode: "Off" — VPA only recommends; you read the numbers and set requests yourself. Great for right-sizing.
  • updateMode: "Auto" — VPA applies changes by evicting and recreating pods.
Rule of thumb: HPA for stateless services that scale by adding replicas; VPA recommendations for right-sizing everything; don't run both in Auto mode on CPU/memory for the same workload — they'll fight.

Cluster Autoscaler

Adding pods only works while nodes have room. The Cluster Autoscaler (or its modern successor Karpenter on AWS) closes the loop at the infrastructure level:

  • Pods that can't be scheduled (state Pending, reason Unschedulable) trigger a new node from the cloud provider.
  • Nodes that stay underutilized get drained and removed.

The full chain during a traffic spike: HPA adds pods → some pods go Pending → Cluster Autoscaler adds a node → scheduler places the pods → traffic is absorbed. Each step typically takes seconds to a couple of minutes.

Keeping scaling safe: PodDisruptionBudget

Scale-downs, node drains, and cluster upgrades all evict pods. A PodDisruptionBudget (PDB) caps how many can be voluntarily disrupted at once:

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-pdb
spec:
  minAvailable: 2          # or maxUnavailable: 1
  selector:
    matchLabels:
      app: web

With this in place, a node drain will wait rather than take your service below two ready replicas.

Load-test it yourself

# Generate load against the web service from a throwaway pod
kubectl run load --image=busybox:1.36 --rm -it --restart=Never -- \
  sh -c 'while true; do wget -q -O- http://web; done'

# In another terminal, watch the HPA react
kubectl get hpa web --watch
kubectl get pods -l app=web --watch

Command recap

kubectl scale deployment/web --replicas=5
kubectl autoscale deployment web --min=2 --max=15 --cpu-percent=70   # quick HPA
kubectl get hpa
kubectl describe hpa web        # events explain every scaling decision
kubectl top pods                # current usage vs requests
kubectl top nodes
