kubectl Cheat Sheet

The commands you'll actually use, grouped by task. Every code block has a copy button — bookmark this page.

Contexts & configuration

kubectl config get-contexts                  # clusters kubectl knows about
kubectl config use-context my-cluster        # switch cluster
kubectl config current-context
kubectl config set-context --current --namespace=staging   # default namespace
kubectl cluster-info
kubectl version
Quality of life: alias it (alias k=kubectl), enable shell completion (kubectl completion bash|zsh|powershell), and install kubectx/kubens for fast context and namespace switching.

Viewing resources

kubectl get pods                             # in current namespace
kubectl get pods -A                          # all namespaces
kubectl get pods -o wide                     # + node, IP
kubectl get pods -w                          # watch live updates
kubectl get pods -l app=web                  # filter by label
kubectl get pods --field-selector=status.phase=Running
kubectl get all                              # pods, services, deployments...
kubectl api-resources                        # every resource type + short names

Inspecting a resource

kubectl describe pod web-7c9d8f6b5-2xkq9     # events at the bottom = gold
kubectl get pod web-7c9d8f6b5-2xkq9 -o yaml  # the full object
kubectl get deploy web -o jsonpath='{.spec.replicas}'
kubectl get pods -o custom-columns=NAME:.metadata.name,NODE:.spec.nodeName
kubectl explain deployment.spec.strategy     # built-in field documentation

Creating & updating

kubectl apply -f app.yaml                    # create or update (idempotent)
kubectl apply -f manifests/                  # a whole directory
kubectl diff -f app.yaml                     # preview what apply would change
kubectl delete -f app.yaml
kubectl edit deployment web                  # edit live in $EDITOR

# Generate YAML instead of writing from scratch:
kubectl create deployment web --image=nginx:1.27 --dry-run=client -o yaml > deploy.yaml
kubectl create service clusterip web --tcp=80:8080 --dry-run=client -o yaml
Habit worth building: prefer declarative apply with YAML files in Git over imperative commands. The imperative forms are still perfect for generating boilerplate with --dry-run=client -o yaml.

Logs & debugging

kubectl logs web-7c9d8f6b5-2xkq9             # container logs
kubectl logs -f deploy/web                   # follow a deployment's logs
kubectl logs web-7c9d8f6b5-2xkq9 --previous  # the crashed instance's logs
kubectl logs web-7c9d8f6b5-2xkq9 -c sidecar  # specific container

kubectl exec -it web-7c9d8f6b5-2xkq9 -- sh   # shell into a container
kubectl debug -it web-7c9d8f6b5-2xkq9 --image=busybox --target=nginx   # ephemeral debug container
kubectl run tmp --image=busybox --rm -it --restart=Never -- sh         # throwaway pod

kubectl get events --sort-by=.metadata.creationTimestamp
kubectl top pods                             # CPU/memory (needs metrics-server)
kubectl top nodes

Networking

kubectl port-forward svc/web 8080:80         # service → localhost:8080
kubectl port-forward pod/web-abc123 8080:80
kubectl get endpoints web                    # is anything behind the service?
kubectl expose deployment web --port=80 --target-port=8080

Rollouts & scaling

kubectl rollout status deployment/web
kubectl rollout history deployment/web
kubectl rollout undo deployment/web
kubectl rollout restart deployment/web       # bounce all pods
kubectl scale deployment/web --replicas=5
kubectl set image deployment/web nginx=nginx:1.28

Nodes & maintenance

kubectl get nodes -o wide
kubectl describe node <node>                 # capacity, conditions, pods on it
kubectl cordon <node>                        # no new pods
kubectl drain <node> --ignore-daemonsets --delete-emptydir-data   # evict for maintenance
kubectl uncordon <node>                      # back in service
kubectl taint nodes <node> dedicated=gpu:NoSchedule

The debugging decision tree

When "it doesn't work", walk this order:

  1. kubectl get pods — is the pod Running and READY x/x?
  2. Pending? → kubectl describe pod: unschedulable (resources, taints) or volume problems.
  3. ImagePullBackOff? → image name, tag, or registry credentials (imagePullSecrets).
  4. CrashLoopBackOff? → kubectl logs --previous: the app is exiting; fix the app or its config.
  5. Running but unreachable? → kubectl get endpoints: empty means the Service selector doesn't match or readiness probes fail.
  6. Reachable in-cluster but not outside? → Ingress rules, IngressClass, controller logs.

Where to go from here

  • Helm — package manager for Kubernetes; templated, versioned releases of whole app stacks.
  • Kustomize — built into kubectl (kubectl apply -k); patch base YAML per environment without templates.
  • GitOps — Argo CD / Flux keep the cluster synced to a Git repo automatically.
  • Observability — Prometheus + Grafana for metrics, Loki for logs.
  • Official docs — kubernetes.io/docs and the interactive tutorials.

You've covered the full core of Kubernetes — architecture, workloads, networking, configuration, storage, and operations. The best next step: take a small real app and deploy it end-to-end with everything you've learned.
