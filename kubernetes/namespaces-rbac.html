Namespaces & RBAC

Namespaces carve one physical cluster into logical spaces — per team, per environment, per app. RBAC (Role-Based Access Control) then decides who can do what in each space. Together they are how clusters stay organized and secure as they grow.

Namespaces

A namespace scopes resource names (two teams can both have a Deployment called api), and is the unit for access control and resource quotas. Every cluster starts with:

NamespaceContains
defaultWhere things land when you don't specify — fine for learning, avoid in production.
kube-systemKubernetes' own components (CoreDNS, kube-proxy…). Don't touch casually.
kube-publicReadable by everyone, rarely used.
kube-node-leaseNode heartbeat objects.
kubectl create namespace staging
kubectl get pods -n staging              # -n targets a namespace
kubectl get pods -A                      # -A = all namespaces
kubectl config set-context --current --namespace=staging   # change your default
kubectl apply -f app.yaml -n staging
Deleting a namespace deletes everything inside it. kubectl delete namespace staging removes every deployment, service, secret, and claim it contains. Powerful for cleanup, catastrophic by accident.

Cross-namespace DNS works with qualified names: api.staging reaches Service api in namespace staging. Note that namespaces do not block network traffic by default — that requires NetworkPolicies.

Quotas and limits per namespace

apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: staging
spec:
  hard:
    requests.cpu: "10"
    requests.memory: 20Gi
    pods: "50"
---
apiVersion: v1
kind: LimitRange
metadata:
  name: defaults
  namespace: staging
spec:
  limits:
    - type: Container
      defaultRequest: { cpu: 100m, memory: 128Mi }
      default:        { cpu: 500m, memory: 512Mi }

ResourceQuota caps a namespace's total consumption; LimitRange fills in requests/limits for containers that forgot to set them — which also keeps the HPA and scheduler working sensibly.

RBAC: who can do what

Every API request is checked against RBAC rules. Four objects define the model, in two pairs:

ObjectScopeDefines
RoleOne namespaceA set of permissions ("verbs on resources").
ClusterRoleWhole clusterSame, but cluster-wide (or reusable across namespaces).
RoleBindingOne namespaceGrants a Role (or ClusterRole) to users/groups/ServiceAccounts in that namespace.
ClusterRoleBindingWhole clusterGrants a ClusterRole everywhere.

Permissions are additive only — there is no "deny" rule; you start with nothing and grant what's needed.

Example: read-only access for a developer

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: staging
rules:
  - apiGroups: [""]                      # "" = the core API group
    resources: ["pods", "pods/log"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: dev-pod-reader
  namespace: staging
subjects:
  - kind: User
    name: jane@example.com
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io

Jane can now read pods and logs in staging — and do nothing else, nowhere else.

Kubernetes ships convenient built-in ClusterRoles you can bind instead of writing your own: view (read-only), edit (read-write, no RBAC changes), admin (namespace admin), cluster-admin (everything).

# Give the frontend team edit rights in their namespace, one line:
kubectl create rolebinding frontend-edit \
  --clusterrole=edit --group=frontend-team -n frontend

ServiceAccounts: identity for pods

Humans authenticate via your identity provider; pods authenticate as a ServiceAccount. Every pod gets one (default if unspecified). If your app talks to the Kubernetes API — operators, CI runners, controllers — give it its own account with minimal rights:

apiVersion: v1
kind: ServiceAccount
metadata:
  name: log-reader
  namespace: staging
---
# bind it like any subject:
subjects:
  - kind: ServiceAccount
    name: log-reader
    namespace: staging
# and in the pod spec:
spec:
  serviceAccountName: log-reader

Checking permissions

kubectl auth can-i delete pods -n staging                  # can *I*?
kubectl auth can-i list secrets --as=jane@example.com      # can *she*?
kubectl auth can-i get pods \
  --as=system:serviceaccount:staging:log-reader            # can the service account?
kubectl auth can-i --list -n staging                       # everything I can do here

A sane starting layout

  • One namespace per app per environment (shop-staging, shop-prod) — or per team in shared clusters.
  • ResourceQuota + LimitRange on every namespace from day one.
  • Developers get edit in dev/staging namespaces, view in prod; deploys to prod go through CI's ServiceAccount.
  • Nobody uses cluster-admin day-to-day.
