Ingress

Ingress is the standard way to route HTTP and HTTPS traffic from outside the cluster to Services inside it. One entry point handles many services, routed by hostname and path, with TLS termination — instead of one cloud load balancer per service.

Ingress = rules + a controller

This trips up everyone once: an Ingress object is just routing rules. Nothing happens unless an Ingress controller — an actual reverse proxy running in the cluster — reads those rules and implements them. Popular controllers:

  • ingress-nginx — the most widely used, runs anywhere.
  • Traefik — default in k3s, great dev experience.
  • Cloud-native controllers — AWS ALB, GKE Ingress, Azure AGIC: rules become cloud load balancer config.
# minikube: one command
minikube addons enable ingress

# anywhere else: install ingress-nginx (via Helm)
helm upgrade --install ingress-nginx ingress-nginx \
  --repo https://kubernetes.github.io/ingress-nginx \
  --namespace ingress-nginx --create-namespace

A complete Ingress

Route two hostnames and an API path prefix to three different services:

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: main
spec:
  ingressClassName: nginx        # which controller should implement this
  rules:
    - host: shop.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: shop-frontend
                port:
                  number: 80
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: shop-api
                port:
                  number: 8080
    - host: blog.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: blog
                port:
                  number: 80

Traffic flow: client → DNS → Ingress controller's load balancer → controller pod matches host/path → forwards to the Service's pods.

pathType matters

pathTypeBehavior
PrefixMatches by URL path segments: /api matches /api and /api/users, but not /apiv2.
ExactMatches the exact path only.
ImplementationSpecificWhatever the controller decides — avoid unless you need controller-specific matching.

TLS / HTTPS

Ingress terminates TLS using a certificate stored in a Secret:

spec:
  tls:
    - hosts:
        - shop.example.com
      secretName: shop-tls      # kubernetes.io/tls Secret in the same namespace
  rules:
    ...
# From existing cert files:
kubectl create secret tls shop-tls --cert=tls.crt --key=tls.key
In practice nobody manages certs by hand: install cert-manager and it will obtain and renew Let's Encrypt certificates automatically — you just add an annotation like cert-manager.io/cluster-issuer: letsencrypt-prod to the Ingress.

Controller-specific behavior via annotations

Features beyond basic routing (redirects, rewrites, rate limits, body size) are configured with controller-specific annotations. For ingress-nginx:

metadata:
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"        # force HTTPS
    nginx.ingress.kubernetes.io/proxy-body-size: "20m"      # allow uploads
    nginx.ingress.kubernetes.io/rewrite-target: /$2         # strip prefixes
    nginx.ingress.kubernetes.io/limit-rps: "10"             # rate limit
Annotations are not portable. They only work with the controller they're written for — switching controllers means rewriting them. This limitation is a big part of why the Gateway API (the typed, role-oriented successor to Ingress) is gradually replacing it. New greenfield projects should evaluate Gateway API; existing Ingress isn't going anywhere soon.

Trying it locally

# 1. Enable the controller (minikube shown)
minikube addons enable ingress

# 2. Deploy an app + service + ingress with host demo.local
kubectl create deployment demo --image=nginx:1.27
kubectl expose deployment demo --port=80
kubectl create ingress demo --class=nginx --rule="demo.local/*=demo:80"

# 3. Point the hostname at the cluster
echo "$(minikube ip) demo.local" | sudo tee -a /etc/hosts
# Windows: add the line to C:\Windows\System32\drivers\etc\hosts

curl http://demo.local

Command recap

kubectl get ingress
kubectl describe ingress main            # rules, backends, and controller events
kubectl get ingressclass                 # which controllers are installed
kubectl logs -n ingress-nginx deploy/ingress-nginx-controller   # access logs / routing errors
