Pods

The Pod is the smallest deployable unit in Kubernetes — a wrapper around one or more containers that always run together, on the same node, sharing the same network identity and storage. Everything else (Deployments, Jobs, StatefulSets) exists to create and manage pods.

Why pods instead of plain containers?

Kubernetes never runs a bare container; it always runs pods. All containers in a pod share:

  • A network namespace — one IP address, one port space. Containers in the same pod reach each other on localhost.
  • Storage volumes — declared once in the pod and mountable by every container in it.
  • A lifecycle — scheduled together, started together, deleted together.

Most pods hold a single container. Multi-container pods are for tightly coupled helpers — a log shipper, a proxy — never for bundling an app with its database.

Your first pod

apiVersion: v1
kind: Pod
metadata:
  name: web
  labels:
    app: web
spec:
  containers:
    - name: nginx
      image: nginx:1.27
      ports:
        - containerPort: 80
      resources:
        requests:
          cpu: 100m        # 0.1 CPU core reserved for scheduling
          memory: 128Mi
        limits:
          memory: 256Mi    # killed (OOM) if it exceeds this
kubectl apply -f pod.yaml        # create it
kubectl get pods -o wide         # status, IP, node
kubectl describe pod web         # events, probes, mounts — your #1 debugging tool
kubectl logs web                 # container stdout/stderr
kubectl exec -it web -- sh       # shell inside the container
kubectl port-forward pod/web 8080:80   # try it on http://localhost:8080
kubectl delete pod web
Don't create bare pods in production. A pod by itself is not restarted if its node dies and can't be scaled or updated. Real workloads use a Deployment, which manages pods for you. Bare pods are fine for quick experiments and debugging.

Pod lifecycle

PhaseMeaning
PendingAccepted by the cluster, but containers aren't running yet — waiting to be scheduled or pulling images.
RunningBound to a node; at least one container is running.
SucceededAll containers exited with code 0 (typical for Jobs).
FailedAll containers terminated and at least one exited non-zero.
UnknownThe node stopped reporting — usually a node or network failure.

Beyond the phase, watch the container states in kubectl describe. Frequent ones: ImagePullBackOff (wrong image name or missing registry credentials) and CrashLoopBackOff (the container keeps exiting; check kubectl logs --previous).

Health probes

Probes let the kubelet ask your app how it's doing, instead of assuming "process running = healthy":

ProbeQuestion it answersOn failure
livenessProbeIs the app alive, or wedged?Container is restarted.
readinessProbeCan it serve traffic right now?Pod is removed from Service endpoints (no restart).
startupProbeHas it finished booting?Other probes wait; container restarted if it never passes.
    containers:
    - name: api
      image: myorg/api:2.1
      readinessProbe:
        httpGet:
          path: /healthz
          port: 8080
        initialDelaySeconds: 5
        periodSeconds: 10
      livenessProbe:
        httpGet:
          path: /healthz
          port: 8080
        failureThreshold: 3
        periodSeconds: 15
Rule of thumb: always set a readiness probe on anything that serves traffic — it's what makes zero-downtime rolling updates actually zero-downtime. Be more careful with liveness probes: an over-eager one turns a slow dependency into a restart storm.

Init containers and sidecars

Init containers run to completion, in order, before the app containers start — perfect for waiting on a database or running migrations:

spec:
  initContainers:
    - name: wait-for-db
      image: busybox:1.36
      command: ['sh', '-c', 'until nc -z db-service 5432; do sleep 2; done']
  containers:
    - name: app
      image: myorg/app:1.0

Sidecar containers run alongside the main container for its whole life — log shippers, TLS proxies, service-mesh agents. Since Kubernetes 1.29 they are declared as init containers with restartPolicy: Always, which starts them before the app and stops them after it.

Requests and limits

  • requests — what the scheduler reserves for the pod. The pod only lands on a node with this much free capacity.
  • limits — the hard ceiling. Exceeding a memory limit gets the container OOM-killed; exceeding a CPU limit just throttles it.

Setting requests accurately matters more than limits: they drive scheduling, autoscaling math, and eviction order. 500m means half a CPU core; Mi/Gi are mebibytes/gibibytes.

Command recap

kubectl run tmp --image=busybox:1.36 --rm -it --restart=Never -- sh   # throwaway debug pod
kubectl get pod web -o yaml          # full object as stored in the cluster
kubectl logs web -f                  # follow logs live
kubectl logs web --previous          # logs from the crashed instance
kubectl top pod web                  # live CPU/memory (needs metrics-server)
