Services

Pods are ephemeral: they get replaced during every deploy and scale event, and each replacement gets a new IP. A Service gives a set of pods one stable virtual IP and DNS name, and load-balances traffic across whichever pods are healthy right now.

How a Service finds its pods

A Service doesn't hold a list of pods — it holds a label selector. Any pod whose labels match, and whose readiness probe passes, becomes an endpoint automatically. Pods come and go; the Service endpoint list updates itself.

apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  selector:
    app: web            # any ready pod with label app=web
  ports:
    - port: 80          # the Service's own port
      targetPort: 8080  # the containerPort it forwards to
kubectl apply -f service.yaml
kubectl get svc web
kubectl get endpoints web      # the actual pod IPs behind the Service
NAME   TYPE        CLUSTER-IP      PORT(S)   AGE
web    ClusterIP   10.96.114.52    80/TCP    10s

NAME   ENDPOINTS                                      AGE
web    10.244.1.5:8080,10.244.1.6:8080,10.244.2.3:8080   10s
Debugging trick: a Service with no endpoints (<none>) is the most common "my app is unreachable" cause. It means the selector matches no pods — or the pods aren't ready. Compare kubectl get pods --show-labels against the Service selector.

Service DNS

CoreDNS gives every Service a predictable name:

<service>.<namespace>.svc.cluster.local

web                     → same namespace, short name works
web.staging             → Service "web" in namespace "staging"
web.staging.svc.cluster.local   → fully qualified

This is how apps talk to each other in-cluster: your API connects to postgres:5432, not to a pod IP. Configuration stays identical across environments.

The four Service types

TypeReachable fromTypical use
ClusterIPInside the cluster only (default)Internal traffic: app → database, service → service.
NodePortEvery node's IP on a fixed port (30000–32767)Quick external access in dev; building block for external LBs.
LoadBalancerA cloud load balancer with a public IPExposing a service directly on the internet (cloud clusters).
ExternalName—A DNS alias: gives an external hostname a cluster-internal name.
# LoadBalancer: same Service, one extra line
apiVersion: v1
kind: Service
metadata:
  name: web-public
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080

On a cloud provider this provisions a real load balancer and populates EXTERNAL-IP. On minikube, run minikube tunnel to simulate one.

LoadBalancer vs Ingress: each LoadBalancer Service costs one cloud load balancer. If you have many HTTP services, put one Ingress in front instead and route by hostname/path — one load balancer for everything.

Headless Services

Set clusterIP: None and the Service gets no virtual IP; DNS instead returns the individual pod IPs. StatefulSets use this so each pod gets its own stable DNS name (db-0.db.default.svc.cluster.local) — essential for databases where "any replica" is not good enough.

apiVersion: v1
kind: Service
metadata:
  name: db
spec:
  clusterIP: None
  selector:
    app: db
  ports:
    - port: 5432

What implements all this?

The Service IP is virtual — no process listens on it. kube-proxy on every node programs iptables/IPVS rules so a packet sent to 10.96.114.52:80 is rewritten to a randomly chosen ready endpoint like 10.244.1.5:8080. That's also why you can't ping a ClusterIP: only the ports declared in the Service are handled.

Testing services from inside the cluster

# Throwaway pod with curl/nslookup for poking at services
kubectl run debug --image=busybox:1.36 --rm -it --restart=Never -- sh

/ # nslookup web
/ # wget -q -O- http://web
/ # exit

# Or forward the service straight to your laptop
kubectl port-forward svc/web 8080:80    # http://localhost:8080

Command recap

kubectl get svc                          # all services
kubectl describe svc web                 # selector, endpoints, events
kubectl get endpoints web                # who is actually behind it
kubectl expose deployment web --port=80 --target-port=8080   # quick service
kubectl port-forward svc/web 8080:80
