Storage & Volumes

A container's filesystem vanishes when the container restarts. Kubernetes storage is a ladder of abstractions — Volumes, PersistentVolumes, PersistentVolumeClaims, and StorageClasses — that takes you from "scratch space for one pod" to "durable disks provisioned on demand".

Volumes: storage scoped to a pod

A volume is declared in the pod spec and mounted into containers. The two everyday in-line types:

apiVersion: v1
kind: Pod
metadata:
  name: cache-example
spec:
  containers:
    - name: app
      image: myorg/app:1.0
      volumeMounts:
        - name: scratch
          mountPath: /tmp/work
  volumes:
    - name: scratch
      emptyDir:
        sizeLimit: 1Gi
TypeLifetimeUse for
emptyDirDeleted with the podScratch space, caches, sharing files between containers in a pod.
hostPathThe node's diskNode agents needing host files (log collectors). Avoid for apps — ties data to one node and is a security risk.
configMap / secretFollows the source objectInjecting config files (covered in the previous chapter).
persistentVolumeClaimIndependent of the podReal data that must survive — see below.

PersistentVolumes and Claims: decoupling apps from disks

Application authors shouldn't need to know whether storage is an EBS volume, an Azure Disk, or NFS. Kubernetes splits the concern:

PersistentVolume (PV)

  • What: an actual piece of storage in the cluster
  • Created by: an admin, or a provisioner automatically
  • Knows: disk IDs, NFS servers, cloud details

PersistentVolumeClaim (PVC)

  • What: an app's request for storage
  • Created by: you, next to your Deployment
  • Knows: only size, access mode, and class

Kubernetes binds each claim to a matching volume. Your pod references the claim; the claim resolves to whatever disk backs it.

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: standard
  resources:
    requests:
      storage: 10Gi
---
apiVersion: v1
kind: Pod
metadata:
  name: postgres
spec:
  containers:
    - name: postgres
      image: postgres:16
      volumeMounts:
        - name: data
          mountPath: /var/lib/postgresql/data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: postgres-data
kubectl get pvc      # STATUS should be "Bound"
kubectl get pv       # the volume it bound to

Access modes

ModeAbbrev.Meaning
ReadWriteOnceRWORead-write by pods on one node at a time. Block storage (EBS, Azure Disk). The common case.
ReadWriteManyRWXRead-write from many nodes simultaneously. Needs file storage (NFS, EFS, Azure Files).
ReadOnlyManyROXRead-only from many nodes — shared reference data.
Classic pitfall: a Deployment with 2+ replicas mounting one RWO claim. The second pod hangs in Pending/ContainerCreating whenever it lands on a different node than the first, because the disk can only attach to one node.

StorageClasses: disks on demand

Nobody pre-creates PVs by hand in a cloud cluster. A StorageClass describes a kind of storage and the CSI driver that can create it; when a PVC names that class, the provisioner creates the disk and its PV automatically — dynamic provisioning.

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: ebs.csi.aws.com          # the CSI driver
parameters:
  type: gp3
reclaimPolicy: Delete                 # what happens to the disk when the PVC is deleted
volumeBindingMode: WaitForFirstConsumer
  • reclaimPolicy: Delete removes the real disk with the claim; Retain keeps it for manual recovery — use Retain for anything precious.
  • WaitForFirstConsumer delays disk creation until a pod is scheduled, so the disk is created in the same availability zone as the node.
  • kubectl get storageclass shows what your cluster offers; one is marked (default) and is used by PVCs that don't specify a class.

StatefulSets: per-replica storage

A Deployment's replicas share claims you create manually — wrong for databases where each replica needs its own disk. StatefulSets stamp out a claim per pod via volumeClaimTemplates:

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: db
spec:
  serviceName: db
  replicas: 3
  selector:
    matchLabels: { app: db }
  template:
    metadata:
      labels: { app: db }
    spec:
      containers:
        - name: postgres
          image: postgres:16
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: [ReadWriteOnce]
        resources:
          requests:
            storage: 10Gi

This creates data-db-0, data-db-1, data-db-2 — and db-1 always reattaches to data-db-1, even after rescheduling.

Command recap

kubectl get pv,pvc,storageclass
kubectl describe pvc postgres-data      # binding events and provisioning errors
kubectl get events | grep -i volume     # attach/mount failures show up here
# Grow a volume (if the StorageClass allows expansion):
kubectl patch pvc postgres-data -p '{"spec":{"resources":{"requests":{"storage":"20Gi"}}}}'
