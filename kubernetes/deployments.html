Deployments & ReplicaSets

A Deployment is the workhorse of Kubernetes: it keeps N identical pod replicas running, replaces them when they die, and rolls out new versions gradually with one command to undo it all. If you deploy a stateless app to Kubernetes, you almost always do it with a Deployment.

The three-layer stack

Deployments don't manage pods directly — there's a layer in between:

  • Deployment — owns the rollout strategy and revision history.
  • ReplicaSet — owns the replica count; creates or deletes pods to keep exactly N matching its selector.
  • Pods — the actual workload.

Each new version of your pod template gets its own ReplicaSet. A rolling update is just the Deployment scaling the new ReplicaSet up while scaling the old one down — and a rollback is the same dance in reverse. You practically never create ReplicaSets by hand.

A complete Deployment

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web            # must match the pod template labels below
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
          ports:
            - containerPort: 80
          readinessProbe:
            httpGet:
              path: /
              port: 80
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
kubectl apply -f deployment.yaml
kubectl get deploy,rs,pods -l app=web    # see all three layers at once
NAME                  READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/web   3/3     3            3           30s

NAME                            DESIRED   CURRENT   READY   AGE
replicaset.apps/web-7c9d8f6b5   3         3         3       30s

NAME                      READY   STATUS    RESTARTS   AGE
pod/web-7c9d8f6b5-2xkq9   1/1     Running   0          30s
pod/web-7c9d8f6b5-8jl4m   1/1     Running   0          30s
pod/web-7c9d8f6b5-tw6zr   1/1     Running   0          30s

Try the self-healing: kubectl delete pod web-7c9d8f6b5-2xkq9 and watch a replacement appear within seconds. The ReplicaSet controller saw 2 pods where 3 were desired and closed the gap.

Rolling updates

Change the pod template — most often the image — and Kubernetes rolls it out gradually:

# Either edit the YAML and re-apply (preferred, keeps Git as truth)...
kubectl apply -f deployment.yaml

# ...or set the image directly
kubectl set image deployment/web nginx=nginx:1.28

kubectl rollout status deployment/web    # watch it progress

The pace is controlled by the update strategy:

spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # at most 1 extra pod during the rollout
      maxUnavailable: 0    # never drop below the desired replica count
  • RollingUpdate (default) — replace pods a few at a time; zero downtime if you have readiness probes.
  • Recreate — kill everything, then start the new version. Simple, has downtime; needed when two versions can't run side by side (e.g. an exclusive database lock).
Readiness probes make rollouts safe: a new pod only starts receiving traffic — and the rollout only proceeds — once its readiness probe passes. Without one, Kubernetes considers a pod "ready" the moment its process starts.

Rollbacks and history

kubectl rollout history deployment/web             # list revisions
kubectl rollout history deployment/web --revision=2
kubectl rollout undo deployment/web                # back to previous revision
kubectl rollout undo deployment/web --to-revision=1
kubectl rollout restart deployment/web             # recreate all pods (picks up new secrets/config)
kubectl rollout pause deployment/web               # stage several changes, then resume
kubectl rollout resume deployment/web
Tip: annotate your deploys with a reason (kubectl annotate deployment/web kubernetes.io/change-cause="upgrade nginx to 1.28") so rollout history shows something more useful than <none>.

Scaling

kubectl scale deployment/web --replicas=5

Or change replicas: in the YAML and re-apply. For automatic scaling based on load, see Scaling & Autoscaling.

The other workload controllers

Deployments are for stateless, interchangeable replicas. Kubernetes has siblings for other shapes of workload:

ControllerUse it forKey property
DeploymentStateless apps: APIs, web servers, workersReplicas are identical and interchangeable.
StatefulSetDatabases, Kafka, anything with identityStable names (db-0, db-1), stable per-pod storage, ordered startup.
DaemonSetNode agents: log collectors, monitoring, CNIExactly one pod on every (matching) node.
JobRun-to-completion tasks: migrations, batchRetries until N successful completions.
CronJobScheduled tasks: backups, reportsCreates a Job on a cron schedule.
# A minimal CronJob: nightly backup at 02:00
apiVersion: batch/v1
kind: CronJob
metadata:
  name: nightly-backup
spec:
  schedule: "0 2 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: backup
              image: myorg/backup:1.4

Command recap

kubectl get deployments
kubectl describe deployment web        # conditions + events when things go wrong
kubectl rollout status deployment/web
kubectl rollout undo deployment/web
kubectl scale deployment/web --replicas=5
kubectl delete deployment web          # deletes its ReplicaSets and pods too
