ConfigMaps & Secrets

Hard-coding configuration into container images means rebuilding for every environment. ConfigMaps and Secrets externalize configuration — the same image runs in dev, staging, and production, with settings and credentials injected by the cluster at runtime.

ConfigMaps

A ConfigMap is a bag of key-value pairs — flags, URLs, or whole config files:

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_ENV: "production"
  LOG_LEVEL: "info"
  DATABASE_HOST: "postgres.default.svc.cluster.local"
  # a whole file as one key:
  nginx.conf: |
    server {
      listen 80;
      location / { proxy_pass http://api:8080; }
    }

Or create one imperatively:

kubectl create configmap app-config \
  --from-literal=APP_ENV=production \
  --from-literal=LOG_LEVEL=info

kubectl create configmap nginx-conf --from-file=nginx.conf

Consuming a ConfigMap

1. As environment variables — the most common pattern:

spec:
  containers:
    - name: app
      image: myorg/app:1.0
      envFrom:
        - configMapRef:
            name: app-config          # every key becomes an env var
      env:
        - name: DB_HOST               # or cherry-pick a single key
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: DATABASE_HOST

2. As files in a volume — for config files your app reads from disk:

spec:
  containers:
    - name: nginx
      image: nginx:1.27
      volumeMounts:
        - name: conf
          mountPath: /etc/nginx/conf.d
  volumes:
    - name: conf
      configMap:
        name: nginx-conf              # each key becomes a file
Update behavior differs: volume-mounted ConfigMaps refresh automatically within about a minute of a change. Environment variables are set once at container start — after editing the ConfigMap you must restart pods: kubectl rollout restart deployment/app.

Secrets

Secrets carry sensitive values — passwords, tokens, TLS keys. Structurally they're ConfigMaps with different handling:

kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password='S3cure!pass'
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:                          # values are base64-encoded
  username: YWRtaW4=
  password: UzNjdXJlIXBhc3M=
stringData:                    # ...or use stringData and write plain text
  api-key: "sk-live-abc123"
Base64 is encoding, not encryption. Anyone who can read the Secret object can decode it instantly (echo YWRtaW4= | base64 -d). Real protection comes from: (1) RBAC restricting who can get secrets, (2) encryption-at-rest for etcd, and (3) never committing Secret manifests to Git.

Consuming Secrets

Identical to ConfigMaps — env vars or volume files:

      env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: password
      volumeMounts:
        - name: certs
          mountPath: /etc/tls
          readOnly: true
  volumes:
    - name: certs
      secret:
        secretName: shop-tls

Prefer volume mounts over env vars for high-value secrets: environment variables leak easily into logs, crash dumps, and kubectl describe output from child processes.

Built-in Secret types

TypePurpose
OpaqueArbitrary key-value data (default).
kubernetes.io/tlsTLS certificate + key; used by Ingress.
kubernetes.io/dockerconfigjsonRegistry credentials for pulling private images (imagePullSecrets).
kubernetes.io/service-account-tokenAPI tokens for ServiceAccounts.
# Private registry access
kubectl create secret docker-registry regcred \
  --docker-server=ghcr.io --docker-username=me --docker-password=<token>
# then in the pod spec:  imagePullSecrets: [{ name: regcred }]

Production patterns

  • GitOps-safe secrets — never commit raw Secrets. Use Sealed Secrets (encrypted manifests only the cluster can decrypt) or the External Secrets Operator (syncs from Vault / AWS Secrets Manager / Azure Key Vault into cluster Secrets).
  • Immutable configs — add immutable: true to ConfigMaps/Secrets that shouldn't change; protects against accidental edits and improves cluster performance.
  • Versioned names — naming configs app-config-v2 and updating the Deployment reference makes every config change a normal, rollback-able rollout.

Command recap

kubectl get configmaps,secrets
kubectl describe configmap app-config
kubectl get secret db-credentials -o jsonpath='{.data.password}' | base64 -d   # decode a value
kubectl edit configmap app-config
kubectl rollout restart deployment/app     # pick up env-var config changes
