Spring Boot Best Practices

Configuration & Structure

1

Use @ConfigurationProperties over @Value

Configuration

Group related properties into a typed, validated POJO. It gives you autocomplete in IDEs, refactoring safety, and a single source of truth — instead of scattered @Value injections spread across dozens of classes.

// application.yml
// app:
//   cache:
//     ttl-seconds: 300
//     max-size: 1000
//   api:
//     key: ${API_KEY}
//     timeout-ms: 5000

@ConfigurationProperties(prefix = "app")
@Validated
public record AppProperties(
    @Valid CacheProperties cache,
    @Valid ApiProperties api
) {
  public record CacheProperties(
      @Positive int ttlSeconds,
      @Positive int maxSize) {}

  public record ApiProperties(
      @NotBlank String key,
      @Positive long timeoutMs) {}
}

// Enable in @SpringBootApplication or a @Configuration class:
@EnableConfigurationProperties(AppProperties.class)

// Usage — inject the record, not individual @Value strings
@Service
public class CacheService {
  private final AppProperties props;
  CacheService(AppProperties props) { this.props = props; }

  Duration getTtl() {
    return Duration.ofSeconds(props.cache().ttlSeconds());
  }
}
2

Externalise all configuration — never hardcode

Configuration

Credentials, URLs, feature flags, and timeouts must live outside the binary. Use environment variables (12-Factor App), Spring Cloud Config, or Kubernetes secrets — and keep application.yml free of production values.

# application.yml — defaults and structure only
spring:
  datasource:
    url: ${DB_URL:jdbc:postgresql://localhost:5432/mydb}
    username: ${DB_USER:myapp}
    password: ${DB_PASS}   # no default — must be set externally

app:
  api:
    key: ${EXTERNAL_API_KEY}  # will fail fast at startup if missing
3

Use profiles for environment-specific configuration

Configuration

Each environment (dev, test, staging, prod) has different infrastructure addresses and behaviour. Profiles let you switch with a single variable (SPRING_PROFILES_ACTIVE=prod) without code changes.

# application.yml — shared defaults
spring:
  jpa.show-sql: false
logging.level.root: WARN

---
# application-dev.yml
spring:
  config.activate.on-profile: dev
  jpa.show-sql: true
  h2.console.enabled: true
logging.level.com.myapp: DEBUG

---
# application-prod.yml
spring:
  config.activate.on-profile: prod
  jpa:
    hibernate.ddl-auto: validate   # never auto-create in prod
logging.level.root: WARN

Dependency Injection & Design

4

Prefer constructor injection over field injection

Design

Constructor injection makes dependencies explicit, enables immutability, and works without the Spring container in unit tests. Field injection with @Autowired hides dependencies and makes testing painful.

// BAD — field injection
@Service
public class OrderService {
  @Autowired private OrderRepository repo;   // hidden dependency
  @Autowired private PaymentGateway gateway;
}

// GOOD — constructor injection (omit @Autowired when single constructor)
@Service
public class OrderService {
  private final OrderRepository repo;
  private final PaymentGateway gateway;

  public OrderService(OrderRepository repo, PaymentGateway gateway) {
    this.repo = repo;
    this.gateway = gateway;
  }
  // Testable: new OrderService(mockRepo, mockGateway)
}
5

Structure packages by feature, not layer

Design

Layer-based structure (controllers/, services/, repositories/) scatters a single feature across three packages. Feature-based structure keeps all related code together, making it easier to delete, move, or extract a feature as a microservice.

# BAD — by layer
com.myapp/
  controllers/ OrderController, UserController
  services/    OrderService, UserService
  repositories/ OrderRepository, UserRepository

# GOOD — by feature
com.myapp/
  order/
    OrderController.java
    OrderService.java
    OrderRepository.java
    OrderDto.java
    Order.java
  user/
    UserController.java
    UserService.java
    ...
6

Keep controllers thin — delegate to services

Design

Controllers should only handle HTTP concerns: parse requests, call the service, and return a response. Business logic in controllers cannot be reused, tested without a servlet container, or called from messaging consumers.

@RestController
@RequestMapping("/api/orders")
public class OrderController {
  private final OrderService orderService;

  OrderController(OrderService orderService) {
    this.orderService = orderService;
  }

  @PostMapping
  public ResponseEntity<OrderResponse> create(
      @Valid @RequestBody CreateOrderRequest req,
      @AuthenticationPrincipal UserDetails user) {
    // No business logic here — just orchestrate HTTP → service → response
    OrderResponse order = orderService.createOrder(req, user.getUsername());
    return ResponseEntity.status(HttpStatus.CREATED).body(order);
  }
}

REST API Design

7

Use DTOs to decouple the API from the domain

REST

Never expose JPA entities directly from REST endpoints. Entities carry persistence annotations, lazy-loaded collections, and internal state you don't want to serialize. DTOs let you version your API independently of the database schema.

// Entity — internal, never serialized directly
@Entity
public class Order {
  @Id @GeneratedValue private Long id;
  private String customerId;
  private BigDecimal total;
  @Enumerated(EnumType.STRING) private OrderStatus status;
  @OneToMany(mappedBy = "order") private List<OrderItem> items;
}

// DTO — public contract, safe to expose
public record OrderResponse(
    Long id,
    String status,
    BigDecimal total,
    int itemCount
) {
  public static OrderResponse from(Order o) {
    return new OrderResponse(o.getId(), o.getStatus().name(),
        o.getTotal(), o.getItems().size());
  }
}
8

Return ResponseEntity for full HTTP control

REST

Returning a bare object from a controller method locks you into status 200. ResponseEntity gives you control over status codes, headers (Location, ETag, Cache-Control), and conditional responses.

@GetMapping("/{id}")
public ResponseEntity<OrderResponse> getOrder(@PathVariable Long id) {
  return orderService.findById(id)
      .map(ResponseEntity::ok)
      .orElse(ResponseEntity.notFound().build());
}

@PostMapping
public ResponseEntity<OrderResponse> create(@Valid @RequestBody CreateOrderRequest req) {
  OrderResponse created = orderService.create(req);
  URI location = ServletUriComponentsBuilder.fromCurrentRequest()
      .path("/{id}").buildAndExpand(created.id()).toUri();
  return ResponseEntity.created(location).body(created);
}

@DeleteMapping("/{id}")
public ResponseEntity<Void> delete(@PathVariable Long id) {
  orderService.delete(id);
  return ResponseEntity.noContent().build();
}
9

Handle all errors centrally with @ControllerAdvice

REST

Scattered try/catch in controllers produces inconsistent error shapes across your API. A single @RestControllerAdvice translates exceptions into a consistent JSON error response — in one place.

public record ErrorResponse(int status, String error, String message, Instant timestamp) {}

@RestControllerAdvice
public class GlobalExceptionHandler {

  @ExceptionHandler(EntityNotFoundException.class)
  public ResponseEntity<ErrorResponse> notFound(EntityNotFoundException ex) {
    return error(HttpStatus.NOT_FOUND, ex.getMessage());
  }

  @ExceptionHandler(MethodArgumentNotValidException.class)
  public ResponseEntity<ErrorResponse> validationError(MethodArgumentNotValidException ex) {
    String msg = ex.getBindingResult().getFieldErrors().stream()
        .map(f -> f.getField() + ": " + f.getDefaultMessage())
        .collect(Collectors.joining(", "));
    return error(HttpStatus.BAD_REQUEST, msg);
  }

  @ExceptionHandler(AccessDeniedException.class)
  public ResponseEntity<ErrorResponse> forbidden(AccessDeniedException ex) {
    return error(HttpStatus.FORBIDDEN, "Access denied");
  }

  @ExceptionHandler(Exception.class)
  public ResponseEntity<ErrorResponse> fallback(Exception ex) {
    log.error("Unhandled exception", ex);
    return error(HttpStatus.INTERNAL_SERVER_ERROR, "An unexpected error occurred");
  }

  private ResponseEntity<ErrorResponse> error(HttpStatus status, String message) {
    return ResponseEntity.status(status)
        .body(new ErrorResponse(status.value(), status.getReasonPhrase(), message, Instant.now()));
  }
}
10

Validate all input with Bean Validation

REST

Never trust user input. Annotate request DTOs with Bean Validation constraints and add @Valid on the parameter — Spring will reject invalid requests before they reach your service.

public record CreateOrderRequest(
    @NotBlank(message = "Customer ID is required")
    String customerId,

    @NotEmpty(message = "Order must have at least one item")
    @Size(max = 50, message = "Order cannot exceed 50 items")
    List<@Valid OrderItemRequest> items,

    @DecimalMin(value = "0.01", message = "Total must be positive")
    @NotNull BigDecimal total
) {}

public record OrderItemRequest(
    @NotNull Long productId,
    @Min(1) @Max(999) int quantity
) {}

// Controller — @Valid triggers validation before the method body runs
@PostMapping
public ResponseEntity<OrderResponse> create(@Valid @RequestBody CreateOrderRequest req) {
  // req is guaranteed valid here
  return ResponseEntity.status(HttpStatus.CREATED).body(orderService.create(req));
}

Data Access

11

Use Spring Data JPA repository interfaces

Data

Extend JpaRepository (or CrudRepository) to get CRUD, pagination, and sorting for free. Use derived query method names for simple queries and @Query for complex ones — avoid hand-written SQL unless absolutely necessary.

public interface OrderRepository extends JpaRepository<Order, Long> {

  // Derived query — Spring generates the JPQL automatically
  List<Order> findByCustomerIdAndStatus(String customerId, OrderStatus status);

  // Pageable support
  Page<Order> findByCustomerId(String customerId, Pageable pageable);

  // Named query for complex JPQL
  @Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.customerId = :cid AND o.total > :min")
  List<Order> findHighValueOrders(@Param("cid") String customerId, @Param("min") BigDecimal min);

  // Projection — only load what you need
  @Query("SELECT new com.myapp.order.OrderSummary(o.id, o.status, o.total) FROM Order o WHERE o.customerId = :cid")
  List<OrderSummary> findSummariesByCustomer(@Param("cid") String customerId);
}
12

Apply @Transactional at the service layer, not the repository

Data

Transactions should span a unit of business work, which typically involves multiple repository calls. Marking individual repository methods transactional creates many small transactions that can't be composed. The service is the right boundary.

// BAD — single-operation transactions in the repository layer
@Repository
public class OrderRepositoryImpl {
  @Transactional  // too granular
  public Order save(Order order) { ... }
}

// GOOD — service method spans the full business operation in one transaction
@Service
@Transactional   // default for all methods in this class
public class OrderService {
  private final OrderRepository orderRepo;
  private final InventoryRepository inventoryRepo;

  public Order createOrder(CreateOrderRequest req) {
    // Both saves happen in a single transaction — all or nothing
    inventoryRepo.decrementStock(req.items());
    return orderRepo.save(Order.from(req));
  }
}
13

Use @Transactional(readOnly = true) for read operations

Data

Read-only transactions skip dirty checking (Hibernate won't compare snapshot vs. current state for every entity in the session), reducing memory use and CPU overhead. They also signal intent clearly — a read-only method should never mutate.

@Service
@Transactional(readOnly = true)   // default: all queries are read-only
public class OrderQueryService {

  public Optional<OrderResponse> findById(Long id) {
    return orderRepo.findById(id).map(OrderResponse::from);
  }

  public Page<OrderResponse> findByCustomer(String customerId, Pageable page) {
    return orderRepo.findByCustomerId(customerId, page).map(OrderResponse::from);
  }

  // Override with write transaction for the one mutation in this class
  @Transactional
  public void markAsViewed(Long id) {
    orderRepo.incrementViewCount(id);
  }
}
14

Avoid the N+1 query problem

Data

Lazy loading a collection inside a loop issues one query per entity — for 100 orders with items, that's 101 queries. Use JOIN FETCH, @EntityGraph, or batch fetching to load associations in bulk.

// BAD — N+1: one query for orders, then one per order for items
List<Order> orders = orderRepo.findAll();
orders.forEach(o -> processItems(o.getItems()));   // N lazy loads!

// GOOD — @EntityGraph: eager-load only for this query
public interface OrderRepository extends JpaRepository<Order, Long> {
  @EntityGraph(attributePaths = {"items", "items.product"})
  List<Order> findByCustomerId(String customerId);
}

// GOOD — JOIN FETCH in @Query
@Query("SELECT DISTINCT o FROM Order o JOIN FETCH o.items WHERE o.status = :status")
List<Order> findWithItemsByStatus(@Param("status") OrderStatus status);

// GOOD — batch fetching in application.yml (applies globally)
// spring.jpa.properties.hibernate.default_batch_fetch_size: 25
15

Use Flyway or Liquibase for database migrations

Data

Never rely on spring.jpa.hibernate.ddl-auto=update in production. It can't drop columns, rename tables, or migrate data safely. Version-controlled migration scripts let you reproduce any database state and roll back systematically.

-- db/migration/V1__create_orders.sql
CREATE TABLE orders (
    id          BIGSERIAL PRIMARY KEY,
    customer_id VARCHAR(36)    NOT NULL,
    status      VARCHAR(20)    NOT NULL DEFAULT 'PENDING',
    total       NUMERIC(12, 2) NOT NULL,
    created_at  TIMESTAMPTZ    NOT NULL DEFAULT now()
);

-- db/migration/V2__add_order_items.sql
CREATE TABLE order_items (
    id         BIGSERIAL PRIMARY KEY,
    order_id   BIGINT         NOT NULL REFERENCES orders(id),
    product_id BIGINT         NOT NULL,
    quantity   INT            NOT NULL,
    unit_price NUMERIC(10, 2) NOT NULL
);

-- db/migration/V3__add_customer_index.sql
CREATE INDEX idx_orders_customer_id ON orders(customer_id);
16

Paginate large result sets

Data

Returning millions of rows crashes the JVM and the client. Always paginate list endpoints using Spring Data's Pageable/Page and expose page, size, and totalElements in the response.

public record PagedResponse<T>(
    List<T> content,
    int page,
    int size,
    long totalElements,
    int totalPages
) {
  public static <T> PagedResponse<T> from(Page<T> page) {
    return new PagedResponse<>(
        page.getContent(), page.getNumber(), page.getSize(),
        page.getTotalElements(), page.getTotalPages());
  }
}

@GetMapping
public ResponseEntity<PagedResponse<OrderResponse>> list(
    @RequestParam(defaultValue = "0") int page,
    @RequestParam(defaultValue = "20") @Max(100) int size,
    @RequestParam(defaultValue = "createdAt,desc") String sort) {
  Pageable pageable = PageRequest.of(page, size, Sort.by(sort.split(",")));
  Page<OrderResponse> result = orderService.findAll(pageable).map(OrderResponse::from);
  return ResponseEntity.ok(PagedResponse.from(result));
}

Performance & Resilience

17

Cache expensive operations with @Cacheable

Performance

Repeated DB or external API calls with the same parameters waste resources. Spring's @Cacheable/@CacheEvict annotations add caching declaratively — backed by Redis, Caffeine, or any JSR-107 provider.

@Service
public class ProductService {

  @Cacheable(value = "products", key = "#id")
  public Optional<ProductResponse> findById(Long id) {
    return productRepo.findById(id).map(ProductResponse::from);
  }

  @CacheEvict(value = "products", key = "#id")
  @Transactional
  public ProductResponse update(Long id, UpdateProductRequest req) {
    var product = productRepo.findById(id).orElseThrow();
    product.update(req);
    return ProductResponse.from(product);
  }

  @CacheEvict(value = "products", allEntries = true)
  @Scheduled(fixedRate = 3_600_000)   // evict all hourly
  public void evictProductCache() {}
}
18

Configure HikariCP connection pool explicitly

Performance

Spring Boot's default HikariCP settings are conservative (pool size 10). Under load, exhausted connections cause timeouts. Size the pool to CPU cores × 2 + disk spindles (Hikari's recommended formula) and set explicit timeouts.

spring:
  datasource:
    hikari:
      maximum-pool-size: 20          # tune per environment
      minimum-idle: 5
      connection-timeout: 30000      # ms — max wait to acquire connection
      idle-timeout: 600000           # ms — remove idle connections after 10m
      max-lifetime: 1800000          # ms — recycle connections after 30m
      connection-test-query: SELECT 1
      pool-name: MyAppHikariPool

Security

19

Secure endpoints explicitly with Spring Security

Security

Define a SecurityFilterChain bean with explicit rules for every URL pattern. Don't rely on default settings — require authentication by default and grant access by exception, not the other way around.

@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class SecurityConfig {

  @Bean
  public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    return http
        .csrf(AbstractHttpConfigurer::disable)          // disable for stateless JWT APIs
        .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/auth/**").permitAll()
            .requestMatchers("/actuator/health").permitAll()
            .requestMatchers(HttpMethod.GET, "/api/products/**").permitAll()
            .requestMatchers("/api/admin/**").hasRole("ADMIN")
            .anyRequest().authenticated()              // deny everything else
        )
        .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)
        .build();
  }
}
20

Use stateless JWT authentication

Security

HTTP sessions are hard to scale horizontally. JWT tokens carry the authentication payload, are verified on every request without a database lookup, and work naturally with load balancers and multiple instances.

@Component
public class JwtAuthFilter extends OncePerRequestFilter {
  private final JwtService jwtService;
  private final UserDetailsService userDetailsService;

  @Override
  protected void doFilterInternal(HttpServletRequest request,
      HttpServletResponse response, FilterChain chain)
      throws ServletException, IOException {

    String header = request.getHeader("Authorization");
    if (header == null || !header.startsWith("Bearer ")) {
      chain.doFilter(request, response);
      return;
    }
    String token = header.substring(7);
    String username = jwtService.extractUsername(token);
    if (username != null && SecurityContextHolder.getContext().getAuthentication() == null) {
      UserDetails user = userDetailsService.loadUserByUsername(username);
      if (jwtService.isValid(token, user)) {
        var auth = new UsernamePasswordAuthenticationToken(user, null, user.getAuthorities());
        auth.setDetails(new WebAuthenticationDetailsSource().buildDetails(request));
        SecurityContextHolder.getContext().setAuthentication(auth);
      }
    }
    chain.doFilter(request, response);
  }
}
21

Never log sensitive data

Security

Passwords, API keys, credit card numbers, and tokens written to logs are accessible to anyone with log access — including in structured logging pipelines and log aggregation services. Mask or exclude them explicitly.

// BAD
log.debug("Creating user with password: {}", request.getPassword());
log.info("Calling payment API with key: {}", apiKey);

// GOOD — log only safe attributes
log.info("Creating user: username={}", request.getUsername());
log.info("Payment gateway call initiated: orderId={}", orderId);

// GOOD — use a DTO toString that masks sensitive fields
public record CreateUserRequest(String username, String email,
    @JsonIgnore String password) {
  @Override public String toString() {
    return "CreateUserRequest{username='%s', email='%s', password='[REDACTED]'}"
        .formatted(username, email);
  }
}

Async & Scheduling

22

Use @Async with a configured thread pool

Async

Spring's @Async by default uses a SimpleAsyncTaskExecutor that creates a new thread per call — unbounded and dangerous under load. Always configure a ThreadPoolTaskExecutor with sensible limits.

@Configuration
@EnableAsync
public class AsyncConfig {
  @Bean(name = "notificationExecutor")
  public Executor notificationExecutor() {
    var executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(4);
    executor.setMaxPoolSize(16);
    executor.setQueueCapacity(100);
    executor.setThreadNamePrefix("notification-");
    executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    executor.initialize();
    return executor;
  }
}

@Service
public class NotificationService {
  // Runs on the named executor; caller continues immediately
  @Async("notificationExecutor")
  public CompletableFuture<Void> sendEmail(String to, String subject, String body) {
    emailClient.send(to, subject, body);
    return CompletableFuture.completedFuture(null);
  }
}
28

Use @Scheduled for background tasks

Async

Spring's scheduling is ideal for cleanup jobs, cache warm-ups, and report generation. Enable it with @EnableScheduling and configure a dedicated thread pool so scheduled tasks don't block web request handling.

@Configuration
@EnableScheduling
public class SchedulingConfig implements SchedulingConfigurer {
  @Override
  public void configureTasks(ScheduledTaskRegistrar registrar) {
    registrar.setScheduler(scheduledTaskExecutor());
  }
  @Bean(destroyMethod = "shutdown")
  public Executor scheduledTaskExecutor() {
    return Executors.newScheduledThreadPool(4);
  }
}

@Component
public class MaintenanceTasks {

  @Scheduled(cron = "0 0 2 * * *")          // 2am daily
  public void archiveOldOrders() { ... }

  @Scheduled(fixedDelay = 60_000, initialDelay = 30_000)  // every 60s, start after 30s
  public void syncInventory() { ... }

  @Scheduled(fixedRate = 300_000)            // every 5 minutes
  public void refreshCaches() { ... }
}

Testing

23

Use test slices: @DataJpaTest and MockMvc

Testing

Loading the full application context for every test is slow. Spring's slice annotations load only the relevant layer — @DataJpaTest for repositories, @WebMvcTest for controllers — keeping tests fast.

// Repository slice — only JPA infrastructure, uses H2 or Testcontainers
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
class OrderRepositoryTest {
  @Autowired private OrderRepository orderRepo;

  @Test void findsByCustomerAndStatus() {
    orderRepo.save(anOrder("cust-1", OrderStatus.PENDING));
    List<Order> found = orderRepo.findByCustomerIdAndStatus("cust-1", OrderStatus.PENDING);
    assertThat(found).hasSize(1);
  }
}

// Controller slice — only web layer, services are mocked
@WebMvcTest(OrderController.class)
class OrderControllerTest {
  @Autowired private MockMvc mockMvc;
  @MockBean private OrderService orderService;

  @Test void returnsCreatedWithLocation() throws Exception {
    given(orderService.create(any())).willReturn(new OrderResponse(42L, "PENDING", bd("99.99"), 2));
    mockMvc.perform(post("/api/orders")
            .contentType(MediaType.APPLICATION_JSON)
            .content("{\"customerId\":\"c1\",\"total\":99.99,\"items\":[]}"))
        .andExpect(status().isCreated())
        .andExpect(header().exists("Location"))
        .andExpect(jsonPath("$.id").value(42));
  }
}
24

Use @SpringBootTest for integration tests

Testing

Integration tests verify the complete request-to-database flow with real infrastructure (PostgreSQL via Testcontainers). Keep them in a separate test source set and run them in CI — not on every build.

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@Testcontainers
class OrderIntegrationTest {

  @Container
  static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16")
      .withDatabaseName("testdb");

  @DynamicPropertySource
  static void registerProps(DynamicPropertyRegistry registry) {
    registry.add("spring.datasource.url", postgres::getJdbcUrl);
    registry.add("spring.datasource.username", postgres::getUsername);
    registry.add("spring.datasource.password", postgres::getPassword);
  }

  @Autowired private TestRestTemplate rest;

  @Test void createOrderReturns201() {
    var req = new CreateOrderRequest("cust-1", List.of(new OrderItemRequest(1L, 3)), bd("59.99"));
    var response = rest.postForEntity("/api/orders", req, OrderResponse.class);
    assertThat(response.getStatusCode()).isEqualTo(HttpStatus.CREATED);
    assertThat(response.getBody().id()).isPositive();
  }
}

Operations & Observability

25

Expose health checks and metrics via Actuator

Ops

Kubernetes liveness/readiness probes and monitoring systems depend on standardised health endpoints. Spring Boot Actuator provides them out of the box — expose /actuator/health publicly and protect management endpoints from external access.

management:
  endpoints:
    web:
      exposure:
        include: health, info, metrics, prometheus
      base-path: /actuator
  endpoint:
    health:
      show-details: when-authorized   # hide DB details from anonymous callers
      probes.enabled: true            # /health/liveness and /health/readiness
  server:
    port: 8081                        # serve actuator on a separate port
    address: 127.0.0.1               # not accessible externally
// Custom health indicator
@Component
public class ExternalApiHealthIndicator implements HealthIndicator {
  private final ExternalApiClient client;

  @Override
  public Health health() {
    try {
      client.ping();
      return Health.up().withDetail("response", "ok").build();
    } catch (Exception e) {
      return Health.down().withDetail("error", e.getMessage()).build();
    }
  }
}
26

Configure CORS explicitly — never use wildcard in production

Security

A wildcard allowedOrigins("*") allows any origin to make authenticated cross-origin requests. Specify exact origins for each environment and ensure preflight caching is configured.

@Configuration
public class CorsConfig {

  @Value("${app.cors.allowed-origins}")
  private List<String> allowedOrigins;

  @Bean
  public CorsConfigurationSource corsConfigurationSource() {
    var config = new CorsConfiguration();
    config.setAllowedOrigins(allowedOrigins);   // e.g. ["https://app.example.com"]
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE", "OPTIONS"));
    config.setAllowedHeaders(List.of("Authorization", "Content-Type", "X-Request-ID"));
    config.setExposedHeaders(List.of("Location", "X-Total-Count"));
    config.setAllowCredentials(true);
    config.setMaxAge(3600L);                    // cache preflight for 1 hour
    var source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/api/**", config);
    return source;
  }
}
27

Package as a fat JAR and use multi-stage Docker builds

Ops

Spring Boot's fat JAR is the deployment unit — it contains the app and all dependencies. Use a multi-stage Dockerfile to keep the final image small (JRE only) and layer the JAR for fast Docker layer caching.

# Stage 1 — extract layers for better cache utilisation
FROM eclipse-temurin:21-jdk AS builder
WORKDIR /app
COPY target/*.jar app.jar
RUN java -Djarmode=layertools -jar app.jar extract

# Stage 2 — runtime image (JRE only, no build tools)
FROM eclipse-temurin:21-jre
WORKDIR /app
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser
USER appuser

COPY --from=builder /app/dependencies/ ./
COPY --from=builder /app/spring-boot-loader/ ./
COPY --from=builder /app/snapshot-dependencies/ ./
COPY --from=builder /app/application/ ./

EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=5s \
  CMD wget -qO- http://localhost:8080/actuator/health || exit 1
ENTRYPOINT ["java", "-XX:MaxRAMPercentage=75.0", "org.springframework.boot.loader.launch.JarLauncher"]
Spring Boot checklist for production: (1) ddl-auto: validate — never update or create-drop. (2) All secrets from environment variables. (3) Actuator on a separate port. (4) HikariCP pool sized explicitly. (5) CORS locked to known origins. (6) @Transactional(readOnly=true) on all read-only service methods. (7) Global exception handler returning consistent JSON errors. (8) Integration tests with Testcontainers hitting a real database.
