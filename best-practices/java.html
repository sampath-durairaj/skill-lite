Java Best Practices

25 battle-tested rules for writing Java code that is readable, maintainable, safe, and performant — with concrete examples of what to do and what to avoid.

01

Follow Naming Conventions

Readability

Consistent naming makes code self-documenting. Java has industry-wide conventions that every developer expects — violating them creates cognitive friction for everyone reading your code.

// ✗ Bad — inconsistent, cryptic names
class account_manager { }
int X = 10;
void DoSomething() { }
final int max_retry = 3;

// ✓ Good — standard Java conventions
class AccountManager { }           // PascalCase for classes
int itemCount = 10;                // lowerCamelCase for variables
void processPayment() { }         // lowerCamelCase for methods
static final int MAX_RETRY = 3;   // SCREAMING_SNAKE_CASE for constants
interface Printable { }           // PascalCase for interfaces (no "I" prefix)
enum OrderStatus { PENDING, SHIPPED, DELIVERED }  // PascalCase enum, UPPER values
02

Use Meaningful, Pronounceable Names

Readability

Names should reveal intent. If you need a comment to explain a variable, the name is wrong. Avoid abbreviations, single letters (except loop counters), and vague terms like data, info, or temp.

// ✗ Bad
int d;               // elapsed time in days
List<int[]> list1;
String s = getUserInput();
void calc(int a, int b) { }

// ✓ Good
int elapsedDays;
List<int[]> flaggedCells;
String rawUserInput = getUserInput();
void calculateMonthlyInterest(int principal, int ratePercent) { }

// ✓ Boolean names should read as questions
boolean isEmpty;            // not "empty"
boolean hasPermission;      // not "permission"
boolean isUserAuthenticated; // not "auth"
03

Program to Interfaces, Not Implementations

Design

Declaring variables and parameters with interface types decouples your code from specific implementations, making it easier to swap, mock, and test.

// ✗ Bad — locked to ArrayList
ArrayList<String> names = new ArrayList<>();
HashMap<String, User> cache = new HashMap<>();

// ✓ Good — flexible, mockable
List<String> names = new ArrayList<>();
Map<String, User> cache = new HashMap<>();

// ✓ Same principle for method signatures
// ✗ Bad
public void process(ArrayList<Order> orders) { }

// ✓ Good — accepts any List implementation
public void process(List<Order> orders) { }

// ✓ Even better — accept the most general type needed
public void process(Collection<Order> orders) { }
public void process(Iterable<Order> orders) { }
04

Prefer Immutability

Safety

Immutable objects are inherently thread-safe, easier to reason about, and cannot be corrupted by callers. Make fields final by default and only relax when mutation is truly needed.

// ✗ Bad — mutable, exposes internals
public class Point {
    public int x;
    public int y;
}

// ✓ Good — immutable value object
public final class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public int getX() { return x; }
    public int getY() { return y; }

    // Return new instance instead of mutating
    public Point translate(int dx, int dy) {
        return new Point(x + dx, y + dy);
    }
}

// ✓ Java 16+ — use records for immutable data
public record Point(int x, int y) {
    public Point translate(int dx, int dy) {
        return new Point(x + dx, y + dy);
    }
}
05

Use Optional Instead of Returning null

Safety

null is the billion-dollar mistake. Returning Optional<T> forces callers to explicitly handle the absent case, making NullPointerException much less likely.

// ✗ Bad — caller might forget null check
public User findUserById(long id) {
    return userRepository.find(id);  // can return null
}
User user = findUserById(42);
System.out.println(user.getName()); // NPE if not found

// ✓ Good — absence is explicit in the type
public Optional<User> findUserById(long id) {
    return Optional.ofNullable(userRepository.find(id));
}

// Caller must handle both cases
findUserById(42)
    .map(User::getName)
    .ifPresent(System.out::println);

// Or with a default
String name = findUserById(42)
    .map(User::getName)
    .orElse("Unknown");

// Or throw a meaningful exception
User user = findUserById(42)
    .orElseThrow(() -> new UserNotFoundException("User 42 not found"));
06

Prefer Composition Over Inheritance

Design

Inheritance creates tight coupling between parent and child. Favour composition — hold a reference to a collaborator and delegate to it — for more flexible, reusable designs.

// ✗ Bad — inheriting just to reuse helper logic
class EmailService extends SmtpClient {
    public void sendWelcome(User user) {
        send(user.getEmail(), "Welcome!");  // uses inherited method
    }
}

// ✓ Good — SmtpClient is a collaborator, not a parent
class EmailService {
    private final SmtpClient smtpClient;   // composition

    public EmailService(SmtpClient smtpClient) {
        this.smtpClient = smtpClient;
    }

    public void sendWelcome(User user) {
        smtpClient.send(user.getEmail(), "Welcome!");
    }
}
// Now EmailService can be tested by injecting a mock SmtpClient
07

Always Use try-with-resources

Safety

Any class implementing AutoCloseable (streams, connections, readers) must be closed after use. try-with-resources guarantees closure even when exceptions occur — a manual finally block is error-prone.

// ✗ Bad — resource leak if exception occurs before close()
BufferedReader reader = new BufferedReader(new FileReader("file.txt"));
String line = reader.readLine();
reader.close();   // never reached if readLine() throws

// ✓ Good — auto-closed even on exception
try (BufferedReader reader = new BufferedReader(new FileReader("file.txt"))) {
    String line = reader.readLine();
    System.out.println(line);
}

// ✓ Multiple resources — closed in reverse order
try (
    Connection conn = dataSource.getConnection();
    PreparedStatement stmt = conn.prepareStatement(sql);
    ResultSet rs = stmt.executeQuery()
) {
    while (rs.next()) {
        process(rs.getString("name"));
    }
}
08

Handle Exceptions at the Right Level

Safety

Catch the most specific exception possible. Never swallow exceptions silently. Handle errors where you have enough context to do something meaningful; otherwise let them propagate.

// ✗ Bad — swallows exception, hides bugs
try {
    processPayment(order);
} catch (Exception e) {
    // do nothing
}

// ✗ Bad — catches too broadly
try {
    int result = Integer.parseInt(input);
} catch (Exception e) {
    log.error("Error", e);
}

// ✓ Good — specific catch, meaningful message
try {
    int result = Integer.parseInt(input);
} catch (NumberFormatException e) {
    throw new InvalidInputException("Expected integer, got: " + input, e);
}

// ✓ Good — log + rethrow when you can't fully handle it
try {
    userService.createUser(dto);
} catch (DuplicateEmailException e) {
    log.warn("Duplicate email attempt: {}", dto.getEmail());
    throw e;  // let the controller turn it into a 409
}
09

Use Enums Instead of int Constants

Safety

Integer constants like STATUS_ACTIVE = 1 provide no type safety — any integer can be passed. Enums are compile-time safe, support methods and fields, and are self-documenting.

// ✗ Bad — int constants
public static final int STATUS_PENDING  = 0;
public static final int STATUS_ACTIVE   = 1;
public static final int STATUS_INACTIVE = 2;

void activate(int status) { }   // accepts any int — no safety

// ✓ Good — type-safe enum with behaviour
public enum OrderStatus {
    PENDING("Awaiting payment"),
    PROCESSING("Being prepared"),
    SHIPPED("En route"),
    DELIVERED("Received"),
    CANCELLED("Cancelled");

    private final String description;

    OrderStatus(String description) {
        this.description = description;
    }

    public String getDescription() { return description; }

    public boolean isFinal() {
        return this == DELIVERED || this == CANCELLED;
    }
}

void ship(OrderStatus status) { }  // only OrderStatus values accepted
10

Override equals() and hashCode() Together

Correctness

If you override equals(), you must override hashCode(). Objects that are equals() must have the same hashCode(). Violating this breaks HashMap, HashSet, and any hash-based structure.

// ✗ Bad — only overrides equals, breaks HashMap
public class Product {
    private final String sku;
    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Product p)) return false;
        return Objects.equals(sku, p.sku);
    }
    // Missing hashCode — Product is broken in HashSet/HashMap!
}

// ✓ Good — both overridden consistently
import java.util.Objects;

public class Product {
    private final String sku;
    private final String name;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Product p)) return false;
        return Objects.equals(sku, p.sku);
    }

    @Override
    public int hashCode() {
        return Objects.hash(sku);   // same fields as equals
    }
}

// ✓ Best for pure data — use records (auto-generates both)
public record Product(String sku, String name) { }
11

Use the Builder Pattern for Complex Objects

Design

Constructors with many parameters are hard to read and error-prone — it's easy to swap two arguments of the same type. Builders make construction explicit and readable, especially when some parameters are optional.

// ✗ Bad — what does each argument mean?
new User("Alice", "alice@example.com", true, false, null, "en");

// ✓ Good — self-documenting builder
User user = User.builder()
    .name("Alice")
    .email("alice@example.com")
    .active(true)
    .locale("en")
    .build();

// Classic builder implementation
public class User {
    private final String name;
    private final String email;
    private final boolean active;
    private final String locale;

    private User(Builder b) {
        this.name   = Objects.requireNonNull(b.name, "name required");
        this.email  = Objects.requireNonNull(b.email, "email required");
        this.active = b.active;
        this.locale = b.locale != null ? b.locale : "en";
    }

    public static Builder builder() { return new Builder(); }

    public static class Builder {
        private String name, email, locale;
        private boolean active = true;

        public Builder name(String v)   { this.name = v;   return this; }
        public Builder email(String v)  { this.email = v;  return this; }
        public Builder active(boolean v){ this.active = v;  return this; }
        public Builder locale(String v) { this.locale = v; return this; }
        public User build()             { return new User(this); }
    }
}
12

Validate Parameters at the Boundary

Safety

Validate inputs at the point they enter your system — public constructors, public methods, and API handlers. Failing fast with a clear message is infinitely better than a NullPointerException three stack frames deeper.

import java.util.Objects;

public class BankAccount {
    private final String accountId;
    private double balance;

    public BankAccount(String accountId, double initialBalance) {
        // Validate immediately, fail with meaningful messages
        this.accountId = Objects.requireNonNull(accountId, "accountId must not be null");
        if (accountId.isBlank()) throw new IllegalArgumentException("accountId must not be blank");
        if (initialBalance < 0)  throw new IllegalArgumentException("Initial balance cannot be negative");
        this.balance = initialBalance;
    }

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Deposit amount must be positive, was: " + amount);
        this.balance += amount;
    }

    public void withdraw(double amount) {
        if (amount <= 0)          throw new IllegalArgumentException("Withdrawal amount must be positive");
        if (amount > this.balance) throw new IllegalStateException("Insufficient funds");
        this.balance -= amount;
    }
}
13

Use StringBuilder for String Concatenation in Loops

Performance

Strings are immutable. Using + inside a loop creates a new String object on every iteration — O(n²) allocations. StringBuilder maintains a resizable buffer and is O(n).

// ✗ Bad — O(n²) allocations for n items
String result = "";
for (String item : items) {
    result += item + ", ";  // creates new String each iteration
}

// ✓ Good — single buffer, O(n)
StringBuilder sb = new StringBuilder();
for (String item : items) {
    sb.append(item).append(", ");
}
String result = sb.toString();

// ✓ Even better — use String.join() or Stream for simple cases
String joined = String.join(", ", items);

String joined2 = items.stream()
    .collect(java.util.stream.Collectors.joining(", "));

// Note: the "+" operator INSIDE a single expression (not loop) is fine
// The compiler optimises it automatically
String msg = "Hello, " + name + "! You have " + count + " messages."; // OK
14

Use Generics — Avoid Raw Types

Safety

Raw types (e.g. List instead of List<String>) bypass the type system, pushing errors to runtime. Generics catch type mismatches at compile time and eliminate unchecked casts.

// ✗ Bad — raw type, unchecked cast at runtime
List names = new ArrayList();
names.add("Alice");
names.add(42);              // compiles, then blows up
String first = (String) names.get(0);

// ✓ Good — type-safe, compiler catches mistakes
List<String> names = new ArrayList<>();
names.add("Alice");
// names.add(42);           // compile error — caught early
String first = names.get(0); // no cast needed

// ✓ Generic methods
public <T extends Comparable<T>> T max(List<T> list) {
    return list.stream().max(Comparator.naturalOrder())
               .orElseThrow(NoSuchElementException::new);
}

// ✓ Bounded wildcards for flexibility
public double sumList(List<? extends Number> list) {
    return list.stream().mapToDouble(Number::doubleValue).sum();
}
15

Prefer the Stream API for Collection Processing

Design

Stream pipelines express what to compute, not how — making the intent immediately clear. They also enable lazy evaluation and easy parallelism.

// ✗ Imperative — verbose, noisy
List<String> result = new ArrayList<>();
for (Order order : orders) {
    if (order.getStatus() == OrderStatus.SHIPPED) {
        String name = order.getCustomerName().toUpperCase();
        result.add(name);
    }
}
Collections.sort(result);

// ✓ Declarative — reads like a specification
List<String> result = orders.stream()
    .filter(o -> o.getStatus() == OrderStatus.SHIPPED)
    .map(Order::getCustomerName)
    .map(String::toUpperCase)
    .sorted()
    .collect(Collectors.toList());   // or .toList() in Java 16+

// ✓ Grouping, counting
Map<OrderStatus, Long> countByStatus = orders.stream()
    .collect(Collectors.groupingBy(Order::getStatus, Collectors.counting()));

// ✓ Parallel for large datasets (only when order doesn't matter)
long expensiveOrderCount = orders.parallelStream()
    .filter(o -> o.getTotal() > 1000)
    .count();
16

Use Immutable Collections

Safety

Returning mutable internal collections exposes your object's state. Return immutable views or copies so callers cannot corrupt your internals.

// ✗ Bad — caller can modify your internal list
public List<String> getRoles() {
    return roles;   // direct reference leaks
}

// ✓ Good — unmodifiable view (Java 9+)
public List<String> getRoles() {
    return Collections.unmodifiableList(roles);
}

// ✓ Best — factory methods create truly immutable collections (Java 9+)
List<String> DAYS = List.of("Mon", "Tue", "Wed", "Thu", "Fri");
Set<String> CODES = Set.of("A", "B", "C");
Map<String, Integer> HTTP = Map.of("OK", 200, "NOT_FOUND", 404);

// ✓ Copying a mutable collection to make it immutable
List<String> snapshot = List.copyOf(mutableList);

// Note: List.of() / Set.of() do NOT allow null elements
// Use Collections.unmodifiableList(list) if nulls are possible
17

Use Records for Data Carriers (Java 16+)

Modern Java

Records eliminate the boilerplate of DTOs and value objects — constructor, getters, equals(), hashCode(), and toString() are all generated automatically.

// ✗ Bad — 40 lines of boilerplate for a simple DTO
public class UserDto {
    private final String name;
    private final String email;
    public UserDto(String name, String email) { ... }
    public String getName() { return name; }
    public String getEmail() { return email; }
    @Override public boolean equals(Object o) { ... }
    @Override public int hashCode() { ... }
    @Override public String toString() { ... }
}

// ✓ Good — 1 line, all of the above generated
public record UserDto(String name, String email) { }

// ✓ Records with custom validation
public record Temperature(double value, String unit) {
    // Compact constructor — runs before fields are assigned
    public Temperature {
        if (!Set.of("C", "F", "K").contains(unit))
            throw new IllegalArgumentException("Unknown unit: " + unit);
        if (unit.equals("K") && value < 0)
            throw new IllegalArgumentException("Kelvin cannot be negative");
    }

    public Temperature toCelsius() {
        return switch (unit) {
            case "C" -> this;
            case "F" -> new Temperature((value - 32) * 5 / 9, "C");
            case "K" -> new Temperature(value - 273.15, "C");
            default  -> throw new IllegalStateException();
        };
    }
}

Use var for Local Variable Type Inference (Java 10+)

Modern Java

var removes redundant type declarations without sacrificing type safety. Use it when the type is obvious from the right-hand side; keep explicit types when they add clarity.

// ✓ Good uses of var — type is obvious from RHS
var names   = new ArrayList<String>();
var counter = Map.entry("hits", 42);
var path    = Paths.get("/tmp/output.csv");

try (var reader = new BufferedReader(new FileReader("data.txt"))) {
    var line = reader.readLine();
}

// ✓ Great in for-each when type is long
for (var entry : userPermissionMap.entrySet()) {
    System.out.println(entry.getKey() + " → " + entry.getValue());
}

// ✗ Bad uses — type is not obvious; reduces readability
var result = process(data);       // what type does process() return?
var x = 42;                       // just write int x = 42

// var is only for LOCAL variables — cannot use for fields, params, or return types
// var names; ← compile error (cannot infer without initialiser)

Use Pattern Matching instanceof (Java 16+)

Modern Java

Pattern matching eliminates the redundant cast after an instanceof check, reducing boilerplate and making the code harder to misread.

// ✗ Old style — check then cast (redundant)
if (shape instanceof Circle) {
    Circle c = (Circle) shape;   // already know it's a Circle!
    return Math.PI * c.getRadius() * c.getRadius();
}

// ✓ Pattern matching — check + bind in one step (Java 16+)
if (shape instanceof Circle c) {
    return Math.PI * c.getRadius() * c.getRadius();
}

// ✓ With sealed classes and switch expressions (Java 21 — switch patterns)
double area = switch (shape) {
    case Circle c    -> Math.PI * c.radius() * c.radius();
    case Rectangle r -> r.width() * r.height();
    case Triangle t  -> 0.5 * t.base() * t.height();
};

// ✓ Pattern matching in equals()
@Override
public boolean equals(Object o) {
    return o instanceof Point p
        && this.x == p.x
        && this.y == p.y;
}
20

Apply the Single Responsibility Principle

Design

A class should have only one reason to change. When a class handles multiple concerns, a change in one concern risks breaking another. Keep classes focused — small, cohesive, and testable.

// ✗ Bad — one class doing everything
class OrderProcessor {
    public void processOrder(Order order) {
        // Validation logic
        if (order.getItems().isEmpty()) throw new InvalidOrderException("...");
        // Persistence logic
        orderRepository.save(order);
        // Email logic
        emailClient.send(order.getCustomerEmail(), "Your order...");
        // Invoice PDF logic
        pdfService.generate(order);
    }
}

// ✓ Good — each class has one responsibility
class OrderValidator {
    public void validate(Order order) { /* validation only */ }
}
class OrderRepository {
    public void save(Order order) { /* persistence only */ }
}
class OrderNotifier {
    public void notifyCustomer(Order order) { /* email only */ }
}
class InvoiceGenerator {
    public void generate(Order order) { /* PDF only */ }
}
class OrderService {
    // Coordinates the workflow — its single responsibility
    public void processOrder(Order order) {
        validator.validate(order);
        repository.save(order);
        notifier.notifyCustomer(order);
        invoiceGenerator.generate(order);
    }
}

Replace Magic Numbers and Strings with Named Constants

Readability

Literal values embedded in code are opaque and fragile — when you need to change a value, you have to find every occurrence. Named constants document intent and centralise change.

// ✗ Bad — what do these numbers mean?
if (account.getBalance() < 500) {
    applyFee(12.50);
}
Thread.sleep(5000);
String url = "https://api.example.com/v2/users";

// ✓ Good — intent is clear, single place to change
private static final double MINIMUM_BALANCE         = 500.00;
private static final double LOW_BALANCE_FEE         = 12.50;
private static final long   POLL_INTERVAL_MS        = 5_000L;
private static final String USER_API_ENDPOINT       = "https://api.example.com/v2/users";

if (account.getBalance() < MINIMUM_BALANCE) {
    applyFee(LOW_BALANCE_FEE);
}
Thread.sleep(POLL_INTERVAL_MS);

// ✓ Use enum for related groups of constants (see Best Practice #09)
// ✓ Use underscore separators in long numeric literals (Java 7+)
long worldPopulation = 8_000_000_000L;
double pi            = 3.141_592_653;
22

Always Use @Override

Safety

The @Override annotation tells the compiler to verify that you are actually overriding a method. Without it, a typo creates a new method instead of overriding the intended one — a silent, hard-to-find bug.

// ✗ Bad — typo creates a NEW method instead of overriding equals()
public class User {
    public boolean equal(Object o) {  // ← typo: "equal" not "equals"
        // This is never called by Java's equality mechanism
        return o instanceof User u && this.id == u.id;
    }
}
user1.equals(user2); // calls Object.equals() — always false (reference equality)!

// ✓ Good — @Override catches the typo at compile time
public class User {
    @Override
    public boolean equals(Object o) { // compiler verifies this overrides Object.equals
        return o instanceof User u && this.id == u.id;
    }

    @Override
    public int hashCode() {
        return Long.hashCode(id);
    }

    @Override
    public String toString() {
        return "User{id=" + id + ", name='" + name + "'}";
    }
}
23

Use a Proper Logging Framework

Observability

System.out.println has no log levels, no timestamps, no caller info, and cannot be disabled in production. SLF4J with Logback/Log4j2 gives you structured, configurable, production-grade logging.

// ✗ Bad
System.out.println("User " + user.getId() + " logged in");
System.err.println("Error processing payment: " + e.getMessage());

// ✓ Good — SLF4J (facade) + Logback/Log4j2 (implementation)
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class PaymentService {
    // One logger per class — use class literal for refactoring safety
    private static final Logger log = LoggerFactory.getLogger(PaymentService.class);

    public void processPayment(Payment payment) {
        log.debug("Processing payment: id={}, amount={}", payment.getId(), payment.getAmount());
        try {
            gateway.charge(payment);
            log.info("Payment successful: id={}", payment.getId());
        } catch (PaymentGatewayException e) {
            // Pass exception as last argument — SLF4J prints the stack trace
            log.error("Payment failed: id={}", payment.getId(), e);
            throw e;
        }
    }
}
// Use parameterised messages — NOT string concatenation
// log.debug("User: " + id);     ← string built even when DEBUG is disabled
// log.debug("User: {}", id);    ← string built only when DEBUG is enabled
24

Write Thread-Safe Code Correctly

Concurrency

Concurrency bugs are among the hardest to reproduce and debug. Use the right tool: java.util.concurrent classes for shared state, immutable objects where possible, and only raw synchronized when nothing else fits.

// ✗ Bad — not thread-safe counter
private int count = 0;
public void increment() { count++; }  // read-modify-write is not atomic

// ✓ Good — AtomicInteger for single-variable counters
private final AtomicInteger count = new AtomicInteger(0);
public void increment() { count.incrementAndGet(); }

// ✓ ConcurrentHashMap instead of HashMap + synchronized
private final Map<String, Session> sessions = new ConcurrentHashMap<>();
// sessions.put / get / remove are thread-safe without external sync

// ✓ BlockingQueue for producer-consumer
private final BlockingQueue<Task> queue = new LinkedBlockingQueue<>(100);
// Producer
queue.put(task);   // blocks if full
// Consumer
Task t = queue.take();  // blocks if empty

// ✓ Use volatile for visibility-only (no compound actions)
private volatile boolean running = true;
public void stop()  { running = false; }
public void run()   { while (running) { /* ... */ } }

// ✓ Prefer higher-level: ExecutorService over raw Thread
ExecutorService pool = Executors.newFixedThreadPool(4);
Future<Result> future = pool.submit(() -> heavyComputation());
Result result = future.get();

Use Optional Methods Correctly — Never Call get() Without isPresent()

Safety

Calling Optional.get() on an empty Optional throws NoSuchElementException, defeating the entire purpose. Always use the safe terminal methods.

Optional<User> optUser = userRepo.findById(id);

// ✗ Bad — NoSuchElementException if empty
User user = optUser.get();

// ✗ Still bad — verbose, same as null check
if (optUser.isPresent()) {
    User user = optUser.get();
}

// ✓ Good — idiomatic Optional usage
// Provide a default value
User user = optUser.orElse(User.ANONYMOUS);

// Compute a default lazily
User user = optUser.orElseGet(() -> userRepo.createGuest());

// Throw a meaningful exception
User user = optUser.orElseThrow(() -> new UserNotFoundException(id));

// Transform and consume
optUser
    .map(User::getEmail)
    .filter(email -> email.endsWith("@company.com"))
    .ifPresent(email -> sendInternalAlert(email));

// Execute action only if present
optUser.ifPresent(u -> auditLog.record(u, "LOGIN"));

// Execute different actions for present/empty (Java 9+)
optUser.ifPresentOrElse(
    u  -> log.info("Found user: {}", u.getName()),
    () -> log.warn("User {} not found", id)
);
26

Close Streams and Connections — Prevent Resource Leaks

Safety

Unclosed streams exhaust file descriptors; unclosed connections exhaust connection pools. Use try-with-resources for everything AutoCloseable and be especially careful with Java 8 Streams that wrap I/O.

// ✗ Bad — FileInputStream not closed on exception
InputStream in = new FileInputStream("file.txt");
processData(in);
in.close();   // skipped if processData() throws

// ✓ Good — always closed
try (InputStream in = new FileInputStream("file.txt")) {
    processData(in);
}

// ✓ Files.lines() returns a Stream that wraps I/O — MUST be closed
try (Stream<String> lines = Files.lines(Paths.get("data.csv"))) {
    lines.filter(l -> l.startsWith("ACTIVE"))
         .forEach(this::processLine);
}

Prefer Switch Expressions Over Switch Statements (Java 14+)

Modern Java

Switch expressions are exhaustive (compiler ensures every case is covered), support arrow syntax (no fall-through), and return a value directly — eliminating common switch-statement bugs.

// ✗ Old switch statement — fall-through bugs, no return value
String label;
switch (day) {
    case MONDAY:
    case TUESDAY:
        label = "Early week";
        break;
    case WEDNESDAY:
        label = "Mid week";
        break;
    // Forgot THURSDAY, FRIDAY, WEEKEND — no compile error!
    default:
        label = "Unknown";
}

// ✓ Switch expression — exhaustive, no fall-through, returns value
String label = switch (day) {
    case MONDAY, TUESDAY             -> "Early week";
    case WEDNESDAY                   -> "Mid week";
    case THURSDAY, FRIDAY            -> "Late week";
    case SATURDAY, SUNDAY            -> "Weekend";
    // Compiler error if any DayOfWeek value is missing
};

// ✓ With yield for multi-statement cases
int score = switch (grade) {
    case "A" -> 4;
    case "B" -> 3;
    case "C" -> {
        log.warn("Low grade recorded");
        yield 2;
    }
    default  -> 0;
};

Use Sealed Classes for Controlled Hierarchies (Java 17+)

Modern Java

Sealed classes restrict which other classes can extend them. This enables exhaustive pattern matching and communicates that the type hierarchy is closed — callers don't need to handle unknown subtypes.

// ✓ Sealed hierarchy — only these three can implement Shape
public sealed interface Shape
    permits Circle, Rectangle, Triangle { }

public record Circle(double radius)              implements Shape { }
public record Rectangle(double width, double height) implements Shape { }
public record Triangle(double base,  double height)  implements Shape { }

// ✓ Compiler verifies exhaustiveness — no default needed
double area = switch (shape) {
    case Circle c    -> Math.PI * c.radius() * c.radius();
    case Rectangle r -> r.width() * r.height();
    case Triangle t  -> 0.5 * t.base() * t.height();
    // No default required — compiler knows all three cases are covered
};

// ✓ Great for domain modelling — e.g. payment result
public sealed interface PaymentResult permits Success, Failure, Pending { }
public record Success(String transactionId, double amount) implements PaymentResult { }
public record Failure(String reason, int errorCode)        implements PaymentResult { }
public record Pending(String referenceId)                  implements PaymentResult { }

// Processing is exhaustive and type-safe
String message = switch (result) {
    case Success s  -> "Charged " + s.amount();
    case Failure f  -> "Failed: " + f.reason();
    case Pending p  -> "Awaiting confirmation: " + p.referenceId();
};
