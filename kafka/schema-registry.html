Schema Registry & Avro

Without a schema contract, a Kafka topic is just bytes — any producer can send anything. Schema Registry enforces that all messages on a topic conform to a registered schema, catching compatibility breaks at produce time before they crash consumers.

Why schema management matters

In a production system with multiple producer and consumer teams:

  • A producer renames a field → consumers break silently or with cryptic errors.
  • A producer adds a required field → old consumers that don't know it fail to deserialize.
  • A producer changes a field type from int to string → consumers get garbage data.

Schema Registry prevents all of these by validating each message's schema against the registered, versioned schema for the topic before the produce call succeeds.

Apache Avro

Avro is the most popular serialization format for Kafka because:

  • Compact binary format — no field names in the payload (schema is separate), so messages are much smaller than JSON.
  • Rich type system — primitives, records, arrays, maps, enums, unions, and logical types (date, timestamp, decimal).
  • Schema evolution rules — adding optional fields is safe; the schema defines explicit compatibility rules.
// Order schema — order.avsc
{
  "type": "record",
  "name": "Order",
  "namespace": "com.example",
  "fields": [
    { "name": "orderId",    "type": "string" },
    { "name": "customerId", "type": "string" },
    { "name": "total",      "type": "double" },
    { "name": "status",     "type": { "type": "enum", "name": "OrderStatus",
                                     "symbols": ["PLACED","PAID","SHIPPED","CANCELLED"] } },
    { "name": "createdAt",  "type": { "type": "long", "logicalType": "timestamp-millis" } },
    { "name": "notes",      "type": ["null", "string"], "default": null }
  ]
}

Schema Registry — how it works

When a producer serializes a record with the Avro serializer:

  1. The serializer hashes the schema and checks its local cache.
  2. If not cached, it calls the Registry's REST API to register the schema (or retrieve the existing schema ID).
  3. If the schema is incompatible with the registered version, the registration fails and the produce call throws an exception.
  4. The message payload is prefixed with a magic byte 0x00 and the 4-byte schema ID, then the Avro binary body.

Consumers call the Registry to fetch the schema by ID embedded in each message, then deserialize.

Running Schema Registry locally

# docker-compose.yml (add to your Kafka setup)
  schema-registry:
    image: confluentinc/cp-schema-registry:7.6.0
    depends_on: [kafka]
    ports:
      - "8081:8081"
    environment:
      SCHEMA_REGISTRY_KAFKASTORE_BOOTSTRAP_SERVERS: kafka:9092
      SCHEMA_REGISTRY_HOST_NAME: schema-registry
      SCHEMA_REGISTRY_LISTENERS: http://0.0.0.0:8081

Schema Registry REST API

# Register a schema
curl -X POST http://localhost:8081/subjects/orders-value/versions \
  -H "Content-Type: application/vnd.schemaregistry.v1+json" \
  -d '{"schema": "{\"type\":\"record\",\"name\":\"Order\",\"fields\":[...]}"}'

# List all subjects
curl http://localhost:8081/subjects

# Get all versions of a subject
curl http://localhost:8081/subjects/orders-value/versions

# Get schema by version
curl http://localhost:8081/subjects/orders-value/versions/1

# Get latest schema
curl http://localhost:8081/subjects/orders-value/versions/latest

# Check compatibility before registering
curl -X POST http://localhost:8081/compatibility/subjects/orders-value/versions/latest \
  -H "Content-Type: application/vnd.schemaregistry.v1+json" \
  -d '{"schema": "..."}'

Compatibility modes

Each subject has a compatibility level that controls what schema changes are allowed:

ModeNew schema must be…Use when…
BACKWARD (default)Readable by the previous version consumerYou deploy consumers before producers
FORWARDReadable by the new version consumer from old dataYou deploy producers before consumers
FULLBoth BACKWARD and FORWARDYou can't guarantee deploy order
BACKWARD_TRANSITIVEReadable by all previous consumersLong-lived data that old consumers must still read
NONENo checksDevelopment only — never production
Safe evolution rules (BACKWARD compatible): you may add optional fields with a default value; you may remove fields that had a default value. You must NOT rename fields, change types, or add required fields without defaults.

Using Avro with Kafka in Java

<!-- Maven dependencies -->
<dependency>
  <groupId>io.confluent</groupId>
  <artifactId>kafka-avro-serializer</artifactId>
  <version>7.6.0</version>
</dependency>
<dependency>
  <groupId>org.apache.avro</groupId>
  <artifactId>avro</artifactId>
  <version>1.11.3</version>
</dependency>
// Avro producer
Properties props = new Properties();
props.put("bootstrap.servers",        "localhost:9092");
props.put("schema.registry.url",      "http://localhost:8081");
props.put("key.serializer",           StringSerializer.class);
props.put("value.serializer",         KafkaAvroSerializer.class);

// Using generated class (from avro-maven-plugin)
Order order = Order.newBuilder()
    .setOrderId("ord-001")
    .setCustomerId("cust-42")
    .setTotal(99.99)
    .setStatus(OrderStatus.PLACED)
    .setCreatedAt(Instant.now().toEpochMilli())
    .build();

producer.send(new ProducerRecord<>("orders", order.getOrderId().toString(), order));

// Avro consumer
props.put("value.deserializer", KafkaAvroDeserializer.class);
props.put("specific.avro.reader", "true"); // use generated class

KafkaConsumer<String, Order> consumer = new KafkaConsumer<>(props);
consumer.subscribe(List.of("orders"));
ConsumerRecords<String, Order> records = consumer.poll(Duration.ofMillis(100));
for (ConsumerRecord<String, Order> r : records) {
    Order o = r.value();
    System.out.println(o.getOrderId() + " total: " + o.getTotal());
}

Schema evolution example

// Version 2 — add optional "region" field (BACKWARD compatible)
{
  "type": "record",
  "name": "Order",
  "namespace": "com.example",
  "fields": [
    { "name": "orderId",    "type": "string" },
    { "name": "customerId", "type": "string" },
    { "name": "total",      "type": "double" },
    { "name": "status",     "type": { "type": "enum", "name": "OrderStatus",
                                     "symbols": ["PLACED","PAID","SHIPPED","CANCELLED"] } },
    { "name": "createdAt",  "type": { "type": "long", "logicalType": "timestamp-millis" } },
    { "name": "notes",      "type": ["null", "string"], "default": null },
    { "name": "region",     "type": ["null", "string"], "default": null }
  ]
}

Old consumers reading v2 messages simply ignore the new region field. New consumers reading v1 messages get null for region (the default). Both directions work — this is BACKWARD compatible.
