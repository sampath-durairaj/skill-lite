Kafka Producers

A Kafka producer is any application that writes events to Kafka topics. The Java producer client handles serialization, partitioning, batching, compression, retries, and delivery guarantees — all configurable to match your throughput and durability requirements.

A minimal Java producer

import org.apache.kafka.clients.producer.*;
import java.util.Properties;

Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("key.serializer",   "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");

KafkaProducer<String, String> producer = new KafkaProducer<>(props);

// Fire-and-forget (async)
producer.send(new ProducerRecord<>("orders", "order-123", "{\"total\":99.99}"));

// With callback to detect errors
producer.send(new ProducerRecord<>("orders", "order-456", "{\"total\":49.00}"),
    (metadata, exception) -> {
        if (exception != null) {
            exception.printStackTrace();
        } else {
            System.out.printf("Sent to %s partition %d offset %d%n",
                metadata.topic(), metadata.partition(), metadata.offset());
        }
    });

producer.flush();
producer.close();

Acknowledgment levels (acks)

The acks setting controls how many broker acknowledgments the producer waits for before considering a send successful.

acks valueDurabilityThroughputWhen to use
0None — fire and forgetHighestMetrics, logs where occasional loss is acceptable
1Leader onlyHighGeneral purpose, balanced default
all (or -1)All in-sync replicasLower latency cost
Production recommendation: use acks=all with min.insync.replicas=2 and enable.idempotence=true for exactly-once delivery guarantees.

Partitioning strategy

How an event is assigned to a partition depends on its key:

  • Key present — the default partitioner hashes the key with murmur2 and takes the result modulo the partition count. All events with the same key always go to the same partition, preserving order per key.
  • Key null — since Kafka 2.4 the sticky partitioner is used: events accumulate in one partition until the batch is full or linger.ms fires, then the next batch picks a different partition. This maximises batching efficiency.
  • Custom partitioner — implement org.apache.kafka.clients.producer.Partitioner and set partitioner.class.
// Explicit partition targeting
producer.send(new ProducerRecord<>(
    "orders",   // topic
    2,          // partition (overrides partitioner)
    "order-789",// key
    "{\"total\":15.00}" // value
));

Batching and compression

The producer does not send each record individually. It accumulates records in an in-memory buffer per partition and sends them as a batch. Two settings control this:

ConfigDefaultEffect
batch.size16 384 bytesMax batch size. Larger → fewer requests, higher throughput, more memory.
linger.ms0How long to wait for more records before sending. 0 sends immediately; 5–20 ms improves batching at small latency cost.
buffer.memory33 554 432 (32 MB)Total memory for all partition buffers. Blocks if full.
compression.typenoneApply snappy, lz4, zstd, or gzip per batch.
props.put("batch.size",        "65536");    // 64 KB
props.put("linger.ms",         "10");       // wait up to 10 ms
props.put("compression.type",  "lz4");     // fast compression
props.put("buffer.memory",     "67108864"); // 64 MB
Compression tip: lz4 is the best default — very fast CPU-wise with ~40% size reduction. Use zstd when CPU is not a bottleneck and you want the best compression ratio.

Idempotent and transactional producers

Idempotent producer (exactly-once per partition)

Setting enable.idempotence=true assigns a Producer ID (PID) and sequence number to each record. If a broker receives a duplicate (e.g., after a retry), it deduplicates it automatically. This gives you exactly-once delivery per partition with zero code changes.

props.put("enable.idempotence", "true");
props.put("acks",               "all");   // required by idempotence
props.put("retries",            "2147483647");

Transactional producer (exactly-once across partitions)

For atomically writing to multiple topics — or for read-process-write pipelines — use transactions.

props.put("enable.idempotence",   "true");
props.put("transactional.id",     "my-service-prod-1"); // unique per producer instance

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
producer.initTransactions();

try {
    producer.beginTransaction();
    producer.send(new ProducerRecord<>("orders",   "o1", "{\"status\":\"placed\"}"));
    producer.send(new ProducerRecord<>("inventory", "p1", "{\"reserved\":1}"));
    producer.commitTransaction();
} catch (Exception e) {
    producer.abortTransaction();
    throw e;
}
