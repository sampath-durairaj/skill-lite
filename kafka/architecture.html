Kafka Architecture

Understanding Kafka's internals — brokers, topics, partitions, replication, and the KRaft consensus protocol — is the foundation for designing reliable, scalable event streaming systems.

Brokers and clusters

A broker is a single Kafka server process. Brokers receive events from producers, store them on disk, and serve them to consumers. A cluster is a group of brokers working together. Clusters are typically run with an odd number of brokers (3, 5, or 7) so that the cluster can tolerate (N-1)/2 broker failures while remaining available.

Each broker is identified by a unique integer node.id. When a producer or consumer connects, it can connect to any broker — called a bootstrap server — and receive metadata about the full cluster from there.

Topics and partitions

A topic is a named, ordered, append-only stream of events. Topics are split into one or more partitions. Each partition is an independent ordered log stored as a set of segment files on one broker's disk.

Topic "orders" with 3 partitions across 3 brokers

  • Partition 0 (leader: Broker 1, replica: Broker 2)
  • Partition 1 (leader: Broker 2, replica: Broker 3)
  • Partition 2 (leader: Broker 3, replica: Broker 1)

Why partitions matter:

  • Parallelism — producers write to multiple partitions concurrently; consumer group members each process a dedicated partition.
  • Scalability — adding partitions (and brokers to hold them) scales throughput linearly.
  • Ordering — ordering is guaranteed within a partition but not across partitions. Use a message key to ensure related events land in the same partition.
# Create a topic with 6 partitions and replication factor 3
kafka-topics.sh --bootstrap-server localhost:9092 \
  --create --topic orders \
  --partitions 6 \
  --replication-factor 3

# Describe topic to see partition/replica assignment
kafka-topics.sh --bootstrap-server localhost:9092 \
  --describe --topic orders

Replication and fault tolerance

Each partition has one leader broker and zero or more follower replicas. Producers and consumers always interact with the leader. Followers passively replicate the leader's log.

The In-Sync Replicas (ISR) set contains all replicas that are fully caught up with the leader. When the leader fails, the controller elects a new leader from the ISR, ensuring no data loss when acks=all is used by producers.

Rule of thumb: with 3 brokers, replication factor 3, and min.insync.replicas=2, the cluster tolerates one broker failure with zero data loss.

Offsets

Every event in a partition is assigned a monotonically increasing integer called an offset. Offsets start at 0 and increase by 1 for each new event. Consumers track their position in a partition by storing their current offset — either automatically (auto-commit) or manually.

Committed offsets are stored in a special internal Kafka topic called __consumer_offsets. This allows a consumer to restart and continue exactly where it left off.

KRaft — Kafka without ZooKeeper

Prior to Kafka 3.3, an Apache ZooKeeper ensemble was required to store cluster metadata (topic configs, partition leaders, ACLs) and to run controller leader election. This added operational complexity and a scalability ceiling.

KRaft (Kafka Raft Metadata mode) replaces ZooKeeper with a built-in Raft consensus protocol. One or more brokers are designated controllers that manage metadata directly in an internal __cluster_metadata topic.

Consumer groups and partition assignment

A consumer group is a logical subscriber composed of one or more consumer instances sharing a group.id. Kafka ensures each partition of a subscribed topic is assigned to exactly one consumer in the group at a time.

This design gives you both load balancing (scale consumers up to the partition count) and broadcast (multiple consumer groups each receive all events independently).

Topic with 4 partitions, two consumer groups

  • Group A (2 consumers): Consumer A1 reads P0 & P1; Consumer A2 reads P2 & P3
  • Group B (1 consumer): Consumer B1 reads all P0–P3 independently
Scaling rule: you can have at most as many active consumers in a group as there are partitions. Extra consumers sit idle. If you need more parallelism, increase partition count first.

Log compaction

By default, Kafka retains events for a fixed time or size. For key-value changelog topics, you can enable log compaction instead: Kafka keeps only the most recent event for each key, discarding older ones. This is how Kafka Streams KTables and the Schema Registry store state.

# Create a compacted topic
kafka-topics.sh --bootstrap-server localhost:9092 \
  --create --topic user-profiles \
  --partitions 3 --replication-factor 3 \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.01 \
  --config segment.ms=10000
