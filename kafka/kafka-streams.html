Kafka Streams

Kafka Streams is a lightweight Java library for building stateful, fault-tolerant stream processing applications. Unlike Apache Flink or Spark Streaming, it runs inside your application process — no separate cluster required — and reads/writes exclusively from Kafka topics.

KStream vs KTable

The two primary abstractions in the Kafka Streams DSL:

KStreamKTable
RepresentsAn unbounded stream of events (append log)A changelog — the latest value per key (like a database table)
New event with key KAppended — doesn't affect prior eventsUpserts — replaces old value for key K
Backed byA Kafka topicA compacted Kafka topic + local RocksDB state store
Use forEvent logs, clickstream, transactionsUser profiles, product catalog, running totals

Building a streaming topology

import org.apache.kafka.streams.*;
import org.apache.kafka.streams.kstream.*;
import java.util.Properties;

Properties props = new Properties();
props.put(StreamsConfig.APPLICATION_ID_CONFIG,     "word-count-app");
props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG,  "localhost:9092");
props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG,
          Serdes.String().getClass());
props.put(StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG,
          Serdes.String().getClass());

StreamsBuilder builder = new StreamsBuilder();

KStream<String, String> textLines = builder.stream("text-input");

KTable<String, Long> wordCounts = textLines
    .flatMapValues(value -> Arrays.asList(value.toLowerCase().split("\\W+")))
    .groupBy((key, word) -> word)
    .count(Materialized.as("word-count-store"));

wordCounts.toStream().to("word-count-output",
    Produced.with(Serdes.String(), Serdes.Long()));

KafkaStreams streams = new KafkaStreams(builder.build(), props);
streams.start();
Runtime.getRuntime().addShutdownHook(new Thread(streams::close));

Aggregations and grouping

KStream<String, Order> orders = builder.stream("orders",
    Consumed.with(Serdes.String(), orderSerde));

// Group by customer ID and compute total spend per customer
KTable<String, Double> spendPerCustomer = orders
    .groupByKey()
    .aggregate(
        () -> 0.0,                                       // initializer
        (customerId, order, total) -> total + order.getTotal(), // aggregator
        Materialized.<String, Double, KeyValueStore<Bytes, byte[]>>
            as("customer-spend")
            .withValueSerde(Serdes.Double())
    );

spendPerCustomer.toStream().to("customer-spend-output",
    Produced.with(Serdes.String(), Serdes.Double()));

Windowing

Windows let you aggregate events over a time boundary. Kafka Streams supports four window types:

Window typeClassDescription
TumblingTimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5))Fixed, non-overlapping intervals
HoppingTimeWindows.ofSizeAndGrace(size, grace).advanceBy(advance)Fixed size, overlapping — each event can be in multiple windows
SlidingSlidingWindows.ofTimeDifferenceWithNoGrace(timeDiff)Windows defined by the time difference between records
SessionSessionWindows.ofInactivityGapWithNoGrace(Duration.ofMinutes(30))Activity-based — gaps of inactivity close a window
// Count orders per 5-minute tumbling window
KTable<Windowed<String>, Long> orderCountPerWindow = orders
    .groupByKey()
    .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5)))
    .count(Materialized.as("order-count-window"));

orderCountPerWindow.toStream()
    .map((windowedKey, count) -> KeyValue.pair(
        windowedKey.key() + "@" + windowedKey.window().start(),
        count))
    .to("order-count-5min");

Stream-table join

KStream<String, Order> orders    = builder.stream("orders");
KTable<String, Customer> customers = builder.table("customers");

// Enrich each order with customer info (join on customerId key)
KStream<String, EnrichedOrder> enriched = orders.join(
    customers,
    (order, customer) -> new EnrichedOrder(order, customer)
);

enriched.to("enriched-orders");

Interactive queries — querying state stores

Kafka Streams persists aggregation results in local state stores (backed by RocksDB). You can query them directly without producing to another topic:

// After streams.start()
ReadOnlyKeyValueStore<String, Double> store = streams.store(
    StoreQueryParameters.fromNameAndType(
        "customer-spend",
        QueryableStoreTypes.keyValueStore()
    )
);

Double total = store.get("customer-42");
System.out.println("Customer 42 total spend: " + total);

// Iterate all entries
try (KeyValueIterator<String, Double> all = store.all()) {
    while (all.hasNext()) {
        KeyValue<String, Double> entry = all.next();
        System.out.println(entry.key + ": " + entry.value);
    }
}
Deployment: Kafka Streams applications scale by running multiple instances with the same application.id. Kafka automatically distributes partitions (and thus state store shards) across instances. Use a REST endpoint on each instance to proxy interactive queries to the correct shard.
