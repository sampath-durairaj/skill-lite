Consumers & Consumer Groups

Kafka consumers read events from partitions using a pull model. Consumer groups coordinate which consumer instance reads which partition, enabling both horizontal scaling and independent multi-subscriber access to the same topic.

Basic consumer poll loop

import org.apache.kafka.clients.consumer.*;
import java.time.Duration;
import java.util.*;

Properties props = new Properties();
props.put("bootstrap.servers",  "localhost:9092");
props.put("group.id",           "order-processor");
props.put("key.deserializer",   "org.apache.kafka.common.serialization.StringDeserializer");
props.put("value.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("auto.offset.reset",  "earliest"); // start from beginning if no committed offset

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(List.of("orders"));

try {
    while (true) {
        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
        for (ConsumerRecord<String, String> record : records) {
            System.out.printf("topic=%s partition=%d offset=%d key=%s value=%s%n",
                record.topic(), record.partition(), record.offset(),
                record.key(), record.value());
        }
    }
} finally {
    consumer.close(); // commits offsets and triggers rebalance
}

Consumer groups

All consumers sharing the same group.id form a consumer group. Kafka distributes partitions among group members so each partition is consumed by exactly one member at a time.

  • If you run fewer consumers than partitions, some consumers handle multiple partitions.
  • If you run consumers equal to partitions, each consumer gets exactly one partition — maximum parallelism.
  • If you run more consumers than partitions, the extras sit idle until a member fails or leaves.

Multiple different consumer groups each receive all events independently — Kafka remembers each group's offset separately. This is ideal for fan-out scenarios (e.g., one group for analytics, another for archiving).

Offset management

An offset represents the position of the next record a consumer will read. Committing an offset tells Kafka "I've processed everything up to here."

Auto-commit (default)

When enable.auto.commit=true (the default), the consumer commits the latest polled offsets every auto.commit.interval.ms (5 000 ms). This is simple but can cause duplicates: if the consumer crashes after polling but before processing, the uncommitted offsets are re-processed on restart.

Manual commit

props.put("enable.auto.commit", "false");

// ... inside the poll loop ...
for (ConsumerRecord<String, String> record : records) {
    process(record); // your business logic
}

// Sync commit — blocks until broker confirms
consumer.commitSync();

// Or async commit — non-blocking
consumer.commitAsync((offsets, exception) -> {
    if (exception != null) log.error("Commit failed", exception);
});

Commit specific offsets

Map<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();
offsets.put(
    new TopicPartition("orders", 0),
    new OffsetAndMetadata(record.offset() + 1) // commit next-to-read offset
);
consumer.commitSync(offsets);

Seeking to a specific offset

// Assign partitions manually (no group rebalancing)
TopicPartition p0 = new TopicPartition("orders", 0);
consumer.assign(List.of(p0));

// Seek to beginning
consumer.seekToBeginning(List.of(p0));

// Seek to end (tail)
consumer.seekToEnd(List.of(p0));

// Seek to specific offset
consumer.seek(p0, 500L);

// Seek to a timestamp
Map<TopicPartition, Long> timestamps = Map.of(p0, System.currentTimeMillis() - 3_600_000);
Map<TopicPartition, OffsetAndTimestamp> result = consumer.offsetsForTimes(timestamps);
consumer.seek(p0, result.get(p0).offset());

Rebalancing

A rebalance is the process of reassigning partitions among consumer group members. It triggers when:

  • A consumer joins the group (new instance started)
  • A consumer leaves the group (graceful close or crash after session timeout)
  • Topic partitions are added

During a rebalance, all consumption pauses until the new assignment is complete. To avoid data loss or duplicates, commit offsets before the rebalance with a ConsumerRebalanceListener:

consumer.subscribe(List.of("orders"), new ConsumerRebalanceListener() {
    @Override
    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {
        consumer.commitSync(); // commit before losing partitions
    }

    @Override
    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {
        // optional: seek or load state for newly assigned partitions
    }
});

Cooperative (incremental) rebalancing

The default eager rebalance revokes all partitions from all members before reassigning. Kafka 2.4 introduced the CooperativeStickyAssignor, which only moves partitions that actually need to change — dramatically reducing disruption in large consumer groups.

props.put("partition.assignment.strategy",
    "org.apache.kafka.clients.consumer.CooperativeStickyAssignor");

Key consumer configs

ConfigDefaultDescription
group.id—Consumer group name (required for subscribe)
auto.offset.resetlatestearliest to replay from start; latest to skip existing messages
enable.auto.committrueSet false for manual control
max.poll.records500Max records returned per poll call
max.poll.interval.ms300 000Max time between polls before consumer is considered dead
session.timeout.ms45 000Heartbeat timeout — if broker hears nothing, consumer is evicted
heartbeat.interval.ms3 000How often consumer sends heartbeat; should be ⅓ of session.timeout.ms
fetch.min.bytes1Minimum bytes broker accumulates before responding; increase for throughput
fetch.max.wait.ms500Max wait if fetch.min.bytes not yet reached
Tuning tip: if your consumer processes records slowly (e.g., calls an external API), increase max.poll.interval.ms to match your worst-case processing time, or reduce max.poll.records so each batch finishes faster.

Monitoring consumer lag

Consumer lag is the difference between the latest offset in a partition and the consumer's committed offset. High lag means the consumer is falling behind producers.

# Check lag for all groups
kafka-consumer-groups.sh --bootstrap-server localhost:9092 --list

# Detailed lag for a specific group
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --group order-processor --describe

# Reset offsets to beginning (useful for replaying)
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --group order-processor --topic orders \
  --reset-offsets --to-earliest --execute
