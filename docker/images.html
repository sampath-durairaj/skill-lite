Images & Dockerfiles

A Docker image is a read-only, layered filesystem snapshot. It contains everything your app needs to run — OS libraries, runtime, dependencies, and your code. Dockerfiles are the recipes that build images step by step.

Image layers

Every image is composed of layers. Each instruction in a Dockerfile creates a new layer that stacks on the previous one using a union filesystem. Layers are immutable and cached — if nothing changed, Docker reuses the cached layer, making rebuilds fast.

Layer 4 (top)

  • Your app code (COPY)

Layer 3

  • App dependencies (RUN npm install)

Layer 2

  • OS packages (RUN apt-get)

Layer 1 (base)

  • FROM node:20-alpine

Dockerfile instructions

InstructionPurpose
FROMSet the base image. Every Dockerfile starts here.
RUNExecute a shell command and commit the result as a new layer.
COPYCopy files/directories from the build context into the image.
ADDLike COPY but also unpacks tar archives and accepts URLs. Prefer COPY.
WORKDIRSet the working directory for subsequent instructions.
ENVSet environment variables that persist at runtime.
ARGDefine a build-time variable (not available at runtime).
EXPOSEDocument which port the container listens on (informational).
USERSet the user/UID for subsequent instructions and the default runtime user.
CMDDefault command when container starts. Overridable by docker run.
ENTRYPOINTThe main executable. Arguments from CMD or CLI are appended.
HEALTHCHECKCommand Docker runs to check container health.
LABELAdd metadata key=value pairs to the image.

A real-world Dockerfile

# ── Stage 1: build ──────────────────────────────────────────
FROM node:20-alpine AS builder
WORKDIR /app

# Copy package files first — cached if unchanged
COPY package*.json ./
RUN npm ci --only=production

# ── Stage 2: runtime ────────────────────────────────────────
FROM node:20-alpine
WORKDIR /app

# Run as non-root user for security
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# Copy only the production dependencies from builder
COPY --from=builder /app/node_modules ./node_modules
COPY --chown=appuser:appgroup . .

ENV NODE_ENV=production
EXPOSE 3000

HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

CMD ["node", "server.js"]

Building an image

# Build from Dockerfile in current directory, tag as myapp:1.0
docker build -t myapp:1.0 .

# Build with a build argument
docker build --build-arg NODE_ENV=production -t myapp:prod .

# Build from a specific Dockerfile path
docker build -f deploy/Dockerfile -t myapp:prod .

# Show the image layers (history)
docker image history myapp:1.0

# Inspect image metadata
docker image inspect myapp:1.0

Layer caching strategy

Docker rebuilds all layers from the first changed instruction downward. Order matters — put slow, rarely-changing steps early and frequently-changing steps last.

Best practice: copy dependency manifests (package.json, requirements.txt, pom.xml) and install dependencies before copying your source code. Dependency installation is slow; source changes are fast and frequent.

Multi-stage builds

Multi-stage builds let you use one stage to compile or build, and a separate minimal stage for the final image. This dramatically reduces image size by excluding build tools, compilers, and intermediate files.

# Go example: build binary in full SDK, ship in scratch
FROM golang:1.22 AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /app ./cmd/server

FROM scratch
COPY --from=builder /app /app
EXPOSE 8080
ENTRYPOINT ["/app"]

.dockerignore

Like .gitignore but for the Docker build context. Prevents unnecessary files from being sent to the daemon, speeding up builds and preventing secrets from leaking into images.

Image management

# List local images
docker images

# Remove an image
docker rmi myapp:1.0

# Remove all dangling (untagged) images
docker image prune

# Remove all unused images
docker image prune -a

# Tag an existing image
docker tag myapp:1.0 myregistry.io/team/myapp:1.0
