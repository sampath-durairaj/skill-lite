Networking

Docker networking controls how containers communicate with each other, the host, and the outside world. Docker ships with several built-in network drivers, each suited to different use cases.

Network drivers

DriverUse caseContainer-to-container
bridgeDefault for standalone containers on a single hostYes (via IP or name on user-defined bridge)
hostRemove network isolation — container uses host's network stackN/A — same stack as host
noneCompletely isolated, no networkingNo
overlayMulti-host networking (Docker Swarm)Yes, across hosts
macvlanAssign a MAC address directly to a containerVia physical network

The default bridge network

When you run a container without specifying a network, it joins the default bridge network (docker0 interface on the host). Containers on this network can communicate by IP, but not by name — DNS is not available on the default bridge.

# Containers on the default bridge can ping each other by IP
docker run -d --name c1 alpine sleep 3600
docker run -d --name c2 alpine sleep 3600

# Find c1's IP
docker inspect c1 --format '{{.NetworkSettings.IPAddress}}'
# 172.17.0.2

# c2 can reach c1 by IP but NOT by name "c1"
docker exec c2 ping 172.17.0.2      # works
docker exec c2 ping c1              # fails on default bridge

User-defined bridge networks (recommended)

Create a custom bridge network to get automatic DNS — containers can reach each other by container name. This is the recommended approach for all multi-container applications.

# Create a user-defined bridge network
docker network create mynet

# Run containers on the custom network
docker run -d --name app --network mynet myimage
docker run -d --name db  --network mynet postgres:16

# Now 'app' can reach 'db' by name
docker exec app ping db       # works!
docker exec app curl db:5432  # works!

Port publishing

By default, container ports are not reachable from the host or outside. Use -p to publish a port from the container to the host.

# Map container port 80 to host port 8080
docker run -d -p 8080:80 nginx

# Map to a specific host interface only (loopback = local only)
docker run -d -p 127.0.0.1:8080:80 nginx

# Publish all EXPOSE'd ports to random high ports
docker run -d -P nginx
docker port nginx   # see the assigned ports

# Multiple ports
docker run -d -p 80:80 -p 443:443 nginx

Container DNS

On user-defined networks, Docker's embedded DNS server resolves container names and network aliases automatically. Each container gets a DNS entry for its --name.

# Add a network alias (a container can have multiple DNS names)
docker run -d \
  --network mynet \
  --network-alias database \
  --name postgres-primary \
  postgres:16

# Other containers can reach it as either "postgres-primary" or "database"
docker exec app psql -h database -U postgres

Network management

# List networks
docker network ls

# Inspect a network (shows connected containers and IP ranges)
docker network inspect mynet

# Connect a running container to an additional network
docker network connect mynet existing-container

# Disconnect
docker network disconnect mynet existing-container

# Remove a network (all containers must be disconnected first)
docker network rm mynet

# Remove all unused networks
docker network prune
Best practice: create a dedicated user-defined bridge network for each application stack. This provides container name DNS resolution and isolates app traffic from other stacks running on the same host.

Host and none networks

# Host network: container shares the host's network namespace
# No port publishing needed; binds directly to host interfaces
docker run -d --network host nginx
# nginx now listens on the host's port 80 directly

# None: completely isolated (no network interface except loopback)
docker run --rm --network none alpine ip addr
