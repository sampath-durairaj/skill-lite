Registry

A container registry is a storage and distribution system for Docker images. You push images to a registry and pull them on any server, CI runner, or Kubernetes cluster. Docker Hub is the default public registry; cloud providers and self-hosted options are also common in production.

Image naming convention

A fully qualified image name follows this pattern:

[registry-host[:port]/][namespace/]repository[:tag|@digest]

# Examples:
nginx                               # Docker Hub, library image, latest tag
nginx:1.27-alpine                   # Docker Hub, specific tag
myuser/myapp:v2.1                   # Docker Hub, user namespace
ghcr.io/myorg/myapp:abc1234        # GitHub Container Registry
123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest  # AWS ECR

Docker Hub

Docker Hub is the default registry. Images without a registry host are pulled from hub.docker.com. Pull rate limits apply to unauthenticated requests (100/6h) and free accounts (200/6h).

# Log in to Docker Hub
docker login

# Pull an image
docker pull nginx:1.27-alpine

# Pull a specific digest (immutable reference — tag can be reassigned)
docker pull nginx@sha256:a4c399...

# Push your image
docker tag myapp:1.0 myusername/myapp:1.0
docker push myusername/myapp:1.0

# Log out
docker logout

Tagging strategy

Tags are mutable pointers. Use a combination of semantic version tags and a moving latest tag (or stable) so consumers can pin exact versions while still easily tracking the newest release.

# Typical CI pipeline tagging
docker build -t myregistry.io/team/myapp:1.4.2 .
docker tag myregistry.io/team/myapp:1.4.2 myregistry.io/team/myapp:1.4
docker tag myregistry.io/team/myapp:1.4.2 myregistry.io/team/myapp:latest

docker push myregistry.io/team/myapp:1.4.2
docker push myregistry.io/team/myapp:1.4
docker push myregistry.io/team/myapp:latest

AWS ECR workflow

# Authenticate (token is valid for 12 hours)
aws ecr get-login-password --region us-east-1 \
  | docker login --username AWS --password-stdin \
    123456789.dkr.ecr.us-east-1.amazonaws.com

# Create a repository (one-time)
aws ecr create-repository --repository-name myapp

# Tag and push
docker tag myapp:1.0 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:1.0
docker push       123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:1.0

GitHub Container Registry (GHCR)

# Create a Personal Access Token (PAT) with "packages:write" scope
# Then log in:
echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin

# Build and push
docker build -t ghcr.io/myorg/myapp:v1.0 .
docker push    ghcr.io/myorg/myapp:v1.0
CI tip: in GitHub Actions, use GITHUB_TOKEN which is automatically available — no PAT required for pushing to GHCR from within Actions.

Running a private registry

The official registry:2 image gives you a self-hosted registry in seconds. Combine it with a reverse proxy (nginx, Traefik) and TLS for production use.

# Start a local registry on port 5000
docker run -d \
  -p 5000:5000 \
  -v registry-data:/var/lib/registry \
  --name registry \
  --restart unless-stopped \
  registry:2

# Push to it
docker tag myapp:1.0 localhost:5000/myapp:1.0
docker push           localhost:5000/myapp:1.0

# Pull from it
docker pull localhost:5000/myapp:1.0

Image scanning

Scan images for known CVEs before pushing to a registry or deploying to production.

# Using Docker Scout (built into Docker Desktop / CLI)
docker scout cves myapp:1.0

# Quick vulnerability summary
docker scout quickview myapp:1.0

# Using Trivy (open-source, popular in CI)
trivy image myapp:1.0
trivy image --severity HIGH,CRITICAL myapp:1.0
