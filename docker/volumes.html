Volumes & Data

Container filesystems are ephemeral — all writes disappear when a container is removed. Docker provides three mechanisms to persist or share data: named volumes, bind mounts, and tmpfs mounts.

The three storage types

TypeManaged byLocationBest for
Named volumeDockerDocker's storage areaDatabases, persistent app data
Bind mountYou (host path)Anywhere on hostDevelopment (live code reload)
tmpfsDockerHost RAM onlySecrets, caches (never on disk)

Named volumes

Named volumes are created and managed by Docker. They live in Docker's storage directory (/var/lib/docker/volumes/ on Linux) and outlive any individual container. Multiple containers can mount the same volume.

# Create a named volume explicitly
docker volume create pgdata

# Use it when running a container
docker run -d \
  -v pgdata:/var/lib/postgresql/data \
  --name db postgres:16

# Mount the same volume in a second container (read-only)
docker run -d \
  -v pgdata:/data:ro \
  --name backup alpine

Bind mounts

Bind mounts map a host directory or file directly into the container. Changes on either side are immediately visible on the other. This is the standard dev workflow — edit code on the host, see it update in the running container.

# Mount current directory as /app in the container
docker run -it \
  -v "$(pwd)":/app \
  -w /app \
  node:20 npm start

# Mount a single config file (read-only)
docker run -d \
  -v "$(pwd)/nginx.conf":/etc/nginx/nginx.conf:ro \
  nginx
Security note: bind mounts give the container full access to the mounted host path. Avoid mounting sensitive directories (like / or $HOME) or running containers as root with bind mounts in production.

tmpfs mounts

tmpfs stores data entirely in the host's RAM. The data is never written to disk and disappears when the container stops. Use it for temporary secrets or caches that must not persist.

# Mount a tmpfs at /tmp inside the container
docker run -d \
  --tmpfs /tmp:size=64m \
  --name app myimage

# Or using --mount syntax (more explicit)
docker run -d \
  --mount type=tmpfs,target=/tmp,tmpfs-size=64m \
  --name app myimage

Volume management

# List all volumes
docker volume ls

# Inspect a volume (see mount point, labels, etc.)
docker volume inspect pgdata

# Remove a specific volume
docker volume rm pgdata

# Remove all unused volumes (not referenced by any container)
docker volume prune

# Back up a volume to a tar archive
docker run --rm \
  -v pgdata:/data \
  -v "$(pwd)":/backup \
  alpine tar czf /backup/pgdata-backup.tar.gz -C /data .

# Restore from backup
docker run --rm \
  -v pgdata:/data \
  -v "$(pwd)":/backup \
  alpine tar xzf /backup/pgdata-backup.tar.gz -C /data

--mount vs -v syntax

Docker supports two syntaxes for mounts. The newer --mount is more verbose but clearer and is recommended for scripts and production use.

# Named volume — both are equivalent
docker run -v pgdata:/var/lib/postgresql/data postgres
docker run --mount type=volume,src=pgdata,target=/var/lib/postgresql/data postgres

# Bind mount — both are equivalent
docker run -v "$(pwd)":/app node:20
docker run --mount type=bind,src="$(pwd)",target=/app node:20
Best practice: use named volumes in production (Docker manages ownership, backup, and migrations) and bind mounts only in local development for live code reloading.
